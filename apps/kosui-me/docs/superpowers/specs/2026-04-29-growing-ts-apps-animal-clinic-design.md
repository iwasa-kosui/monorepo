# 育てるTypeScriptアプリケーション 第1冊「壊れやすいシステムを育てる」設計書

## 概要

kosui.me 上に、サーバサイドTypeScriptの学習教材シリーズ「育てるTypeScriptアプリケーション」を新設する。第1冊として「壊れやすいシステムを育てる: 動物病院の予約・カルテ管理システムで学ぶサーバサイドTypeScript」を執筆する。

形式は chibivue を参考にした章立て + サイドバーの教材で、`posts` とは独立した `books` コンテンツコレクションを kosui-me 内に追加する。コード本体は別リポジトリで管理し、各章の開始/終了状態を Git tag で固定して読者が手元で動かせるようにする。

## 教材コンセプト

「ひとつの動くアプリに、要件が次々と降ってきて、雑な実装が変更耐性を失っていく。それをリファクタしながら原則を獲得する」というリファクタ駆動の学習教材。chibivue が「ライブラリを下のレイヤーから積み上げる」型を取るのに対し、本書は「動くアプリを段階的に育てる」型を取る。

教えたい原則は `functional-ts-principles` スキルプラグインに既に集約されているが、それを読み物として伝えるには「なぜそれが必要か」が腹落ちしにくい。本書は要件追加によって痛みを発生させ、原則を「降ってきた要件への応答」として導入することで、原則の必要性を身体に覚えさせる。

### 題材として動物病院を選んだ理由

動物病院の予約・カルテ管理は、本書が扱いたい原則を**自然に発生させる**題材として優れている。

- **状態遷移が業務上明確**: 予約 → 受付 → 診察 → 会計の流れと、キャンセルの分岐がある。状態遷移バグ（例: 会計済みの来院が診察中に戻る）は業務インパクトが直感的に大きい
- **状態ごとに必要なデータが大きく違う**: 予約段階では来院理由しかないが、診察中には担当獣医、会計後には診断名・処置内容・金額が乗る。Discriminated Union の必要性が明白に出る
- **外部データ取り込みが現実的**: 検査機関の API、ペット保険、フィラリア予防接種台帳など、外部から取り込むデータの境界防御が必要になる
- **ID取り違えの事故が起きやすい**: ペットID、飼い主ID、来院ID、検体ID と、構造的に似た文字列ID が複数並ぶ
- **規制変更に揺さぶられる**: ペット医療情報の取扱いを規律する業界自主規制の更新や、近未来の関連法改正が想定でき、PII / 機微情報の扱いを「コンプライアンス起点で」見直す必然性を作れる

### コーディングエージェント時代の文脈を組み込む

本書は単なる「リファクタの教科書」ではなく、**コーディングエージェント全盛時代に原則を持たない開発がどう破綻するか**を体感させる教材として設計する。

各章は「降ってきた要件を、原則を知らない素朴な指示でAIエージェントに投げるとどうなるか」を冒頭で例示する。エージェントは既存コードのパターンを忠実に踏襲し、`status: string` の組み合わせをさらに増やし、`as` で型を黙らせ、`throw` の経路を増やしていく。**動いてしまう**ため一見問題なく見えるが、次の要件が降ってきた瞬間に破綻が広がる。読者は章を進めるごとに「原則を持たないままエージェントを使うと、雑な実装が高速に量産される」という実感を獲得する。

最終章では `functional-ts-principles` スキルプラグインを導入することで、同じ要件に対してエージェントが原則に従った実装を出すようになることを示し、**読者が獲得した原則は、自分の手で書く力であると同時に、エージェントを使いこなす力でもある**というメッセージで締める。これは `functional-ts-principles` 公開記事で論じた「コーディングエージェントが生むデリバリー速度への関心が高まる中で、品質を担保する原則への関心が相対的に減っている」という問題意識と地続きである。

## 対象読者

- 他言語（Python / Java / Go / Elixir など）の経験を持つ
- TypeScript 自体は何らかの形で書いた経験がある（フロントエンドなど）
- サーバサイドTypeScriptの本番環境レベルでの実践に不安を持つ

KAKEHASHI Tech Blog の「サーバーサイドTypeScriptの型システムをどう教えるか」（2026年3月公開）が論じた「型検査が保証する範囲とランタイムで担保すべき範囲の境界」を、教える側ではなく**学ぶ側が手を動かして気づく**ための教材として位置づける。

## 約束（What this book promises）

> 本書は、他言語の経験を持ちながらサーバサイドTypeScriptの本番環境レベルでの実践に不安を持つエンジニアに向けて、ひとつの動物病院の予約・カルテ管理アプリを段階的にリファクタしながら、関数型ドメインモデリングの原則を**なぜそれが必要か**から腹落ちさせる教材です。
>
> 各章では、降ってきた要件を素朴な指示でAIエージェントに依頼するとどう実装されるかを例示し、原則を持たないまま開発を進めるとなぜ破綻が広がるのかを体感します。完走すると、Discriminated Union による状態モデリング、Result 型と Railway Oriented Programming、Zod による境界防御、Sensitive 型による機微情報保護といった原則を、降ってきた要件に対する応答として自分の手で書けるようになります。読了後、`functional-ts-principles` スキルプラグインを導入することで、獲得した原則をエージェント時代の品質担保の力として継続適用できます。

## Phase 1 章立て

第0章から第6章 + 最終章の8章構成。第7章（ドメインイベント）と第8章（レイヤードアーキテクチャ）は Phase 2 として将来の拡張余地に残す。

### 第0章: 動くけれど壊れやすいアプリ

- **設定**: 「うちのクリニック向けに、予約とカルテをまとめて管理できる最低限のアプリを作って」という素朴な指示でAIエージェントに依頼して生まれた、動く初期実装という物語にする
- 単一の `Appointment` 型に `status: string` と来院・受付・診察・会計のフィールドをすべて詰め込む
- `throw` だらけ、`as Appointment` だらけ、巨大関数
- ロガーに来院記録を診断名・飼い主連絡先ごとまるごと渡している
- 章本文では、その「素朴な依頼」のプロンプト例とエージェントの応答（要約）を載せ、テストが通ること・要件を満たすことを確認する
- **読者の状態**: 「これは普通に動くな」と思える。同時に「自分も似た指示でエージェントに作らせている」という既視感を持つ

### 第1章: 「会計済みの来院が、なぜか診察中に戻った」

- 降ってくる要件: 不正な状態遷移を防ぎたい
- 引き出す原則: Discriminated Union による状態モデリング、純粋関数による状態遷移、`assertNever` による網羅性
- **読者の状態**: 「型は通るのにランタイムで落ちる」を自分のコードで体験する

### 第2章: 「処方薬・所見・次回予約を診察に紐づけたい」

- 降ってくる要件: 状態ごとに必要なデータが違うことが顕在化
- 引き出す原則: Discriminated Union の各バリアントへの固有フィールド、Companion Object パターン
- **読者の状態**: 状態に紐づくデータとロジックの一体管理を理解する

### 第3章: 「外部の検査機関APIから検査結果を取り込みたい」

- 降ってくる要件: 外部APIの戻り値を `as` していたら本番で `undefined` 参照で落ちた
- 引き出す原則: Zod スキーマによる境界防御、Always-Valid Domain Model、`as` から `safeParse` への移行
- **読者の状態**: kakehashi 記事が論じた「型検査の境界」を要件として体験する

### 第4章: 「ペットIDと飼い主IDを取り違えた」

- 降ってくる要件: 別のペットのカルテに、まったく関係ない飼い主の連絡先が紐づいてしまった事故
- 引き出す原則: Branded Type、`as` をブランド型ファクトリ専用に閉じる
- **読者の状態**: 構造的部分型の限界を ID という最小単位で体感する

### 第5章: 「エラー処理が `if/throw` で破綻している」

- 降ってくる要件: バリデーション失敗・状態遷移違反・外部依存エラーが入り乱れ、try/catch のネストが手に負えない
- 引き出す原則: Result 型、Railway Oriented Programming、ドメイン層からの `throw` 排除
- **読者の状態**: エラーをデータとして扱う発想を獲得する

### 第6章: 「ログにペットの病気の情報を出すことが法改正で禁止された」

- 降ってくる要件: 動物医療情報の取扱いに関する規律が強化され、診断名・処置内容・検査結果といった**ペットの疾病情報**をアプリケーションログに残してはならなくなる。インシデントによる発覚ではなく**コンプライアンス要件として降ってくる**ことに本書なりの捻りを置く
- 引き出す原則: 構造的部分型がなぜ機微情報を型レベルで止められないか、Sensitive 型 + ログ redact による多層防御
- **読者の状態**: 型の限界を認めた上で、仕組みで防ぐ設計を獲得する。インシデント対応とは違い、規制起点で「いま流出していないが将来流出させてはならない」という予防的設計が求められる感覚を得る
- **既存ブログ記事との接続**: 詳細は「ログのPII漏洩を防止する」記事へ送り、本書では要点と本書題材における適用、および**人間のPIIではない動物の医療情報にも同じ仕組みが効く**ことに絞る

### 最終章: ふりかえりと、原則を継続適用するために

- 完成形のコードを第0章のコードと並べ、章ごとに何が変わったかを振り返る
- 各章で見せた「素朴にエージェントに依頼した結果」を一覧で振り返り、原則を持たないままエージェントに依存することのリスクを言語化する
- 同じ要件群を `functional-ts-principles` スキルプラグインを導入したエージェントに依頼するとどう変わるかを実演する。Discriminated Union、Result、Zod、Sensitive 型を踏まえた実装が最初から出力されることを示す
- 読者が獲得した原則は、自分の手で書く力であると同時に、エージェントを使いこなす力でもあることを明示してメッセージを締める
- Phase 2 で扱う範囲（ドメインイベント、レイヤードアーキテクチャ）の予告

## 章内テンプレート

各章を同じリズムで読めるよう、7セクション構成とする。コーディングエージェントへの素朴な依頼の体験を毎章に埋め込むのが本書の核なので、独立したセクションとして固定する。

1. **降ってくる要件** — 院長/獣医/受付スタッフ/規制対応 などからの依頼を自然言語で提示。読者が「自分ならどう直すか」を考える時間を持つ
2. **既存コードを動かしてみる** — 章開始タグを `git checkout` してテストを実行し、要件追加前のコードがどう動くかを確認する
3. **素朴にエージェントに依頼してみる** — 要件をそのままAIエージェントに投げた場合に生成されがちな実装を例示する。生成プロンプトと、エージェントが既存パターンを踏襲して書きそうなコード（あるいは実際にClaude Code等で実行した結果）を載せる。**動いてしまう**こと、テストが通ること、しかし型レベルでも構造でも罠を温存していることを示す
4. **何が痛いのか** — 素朴な実装が抱える痛みを言語化する。型は通るのにランタイムで落ちる、状態の組み合わせが指数的に増える、変更箇所が散らばる、機微情報が透過する、など。次の要件が降ってくると何が起きるかも触れる
5. **どう直すか — 原則の紹介** — その章で導入する原則を最小限の例で説明。詳細は外部リンクに送り、本書では「この題材で何が解決されるか」に絞る
6. **リファクタを進める** — 段階的にコードを書き換える。途中の型エラー・テスト結果を交えながら進める。原則を導入したあとに同じ要件をエージェントに依頼するとどう変わるかを併記する章があってもよい
7. **章末の姿と次章への布石** — `git checkout` で章末タグの状態を確認できる。終わりに「次章で扱う痛み」を1段落だけ予告する

エージェント例示の生成方針:

- 各章の素朴依頼は、書籍執筆時に Claude Code / Codex などで実際に再現性が確認できる依頼文として記載する
- 生成結果はリポジトリの `agent-attempts/step-NN/` 配下に保存し、本文ではその要点を引用する。日付・利用モデルを明記する
- モデルや時期によって出力が変わる可能性があるため「2026年X月時点の Claude Sonnet 4.X / Codex で再現した例」と注記する

## コード管理方針

### リポジトリ構成

kosui.me リポジトリとは別の独立したリポジトリでコード本体を管理する。

- 仮リポジトリ名: `iwasa-kosui/growing-ts-apps-animal-clinic`（決定は実装フェーズ）
- スタック: pnpm + TypeScript + Vitest + Hono + Drizzle（あるいは PostgreSQL + 軽量クエリビルダ。第3章の検査結果取り込みを境に必要になる想定）
- 外部依存はモック可能な構成にし、Docker などローカル実行のハードルを最小化する

### Git tag 運用

各章の開始状態と終了状態を Git tag で固定する。

- `step-00-start`, `step-00-end`, `step-01-start`, `step-01-end`, ... `step-final`
- 章本文中で `git checkout step-NN-start` を案内
- 各章末の差分は `git diff step-NN-start..step-NN-end` で見られる

### step ごとの README

各 step の README に、章のあらすじ、テストの動かし方、要件の再掲を載せる。リポジトリ単独で（kosui.me 本文を読まずに）完走可能な状態を維持する。

### エージェント出力アーカイブ

各章で例示する「素朴にエージェントに依頼した実装」の実出力を、リポジトリ内 `agent-attempts/step-NN/` 配下に保存する。プロンプト、利用モデル、実行日、生成されたコード差分を含める。本文ではこれを引用する形で記述し、再現性が失われた場合でも執筆時点の事実を確認できるようにする。

### 修正運用

タグを誤字や説明不足のたびに打ち直すと本文と乖離する。これを防ぐため次の運用とする。

- kosui.me 本文では `step-NN-end` タグの最新版を参照する前提で書く
- 章ごとの大きな書き換えは CHANGELOG で管理する
- タグを再作成（force update）する場合は CHANGELOG に記録する

## kosui-me 側の実装作業

実装は spec 承認後に別 PR で進める。spec 段階ではスコープのみを定義する。

- `apps/kosui-me/src/content.config.ts` に `books` コレクションを追加
- `apps/kosui-me/src/content/books/animal-clinic/` 配下に冊メタ情報と各章 MDX を配置
- `apps/kosui-me/src/pages/books/index.astro` でシリーズ・冊一覧を表示
- `apps/kosui-me/src/pages/books/[book]/index.astro` で冊トップ（はじめに・目次）を表示
- `apps/kosui-me/src/pages/books/[book]/[chapter].astro` で各章を表示
- `apps/kosui-me/src/layouts/BookLayout.astro` を新設し、サイドバー付き2カラムを実装
- `apps/kosui-me/src/components/BookSidebar.astro` で章一覧を表示
- 既存の検索（Pagefind）に books を含めるかは別途判断（含める場合は `data-pagefind-body` の付与）

## Phase 2 として残す範囲

## リスクと懸念

- **第0章の初期コードのリアリティ**: 「素朴な指示でエージェントが生成しそうなコード」と「他言語出身者が TS で書きそうなコード」を重ね合わせる必要がある。執筆前に実際にエージェントへ依頼して出力を観察し、リアリティを担保する
- **エージェント出力の再現性**: モデル更新やエージェント側の改善で、章ごとに想定したエージェント出力が再現しなくなる可能性がある。日付・モデル名・プロンプトを明記し、`agent-attempts/` に実出力を保存することで「教材執筆時点ではこう出た」事実を残す。再現できなくなった場合の改訂運用は CHANGELOG で管理
- **第6章の規制テーマの扱い**: 動物医療情報を直接規律する明確な国内法は本書執筆時点では未整備。本書では「業界自主規制の強化」「近未来の関連法改正」を**書籍内シナリオ**として扱い、特定の法令名を断定しない。技術的な仕組み（Sensitive 型 + ログ redact）の有効性は規制の細部に依存しないため、シナリオが将来現実とずれても本質は揺らがない
- **第6章のPII章と既存ブログ記事の重複**: 詳細は記事に送り、本書では本書題材への適用と本質に絞る
- **章末タグと本文の乖離**: 上記「修正運用」で吸収する
- **完走率**: Phase 1 で8章。各章 30〜45分程度の読了時間を目安に、本文の長さを設計する
- **コードリポジトリの運用負荷**: kosui.me 本文と独立リポジトリの両方を保守する必要がある。初期は更新頻度を抑え、Phase 1 完成までは「本文を書きながら章末タグを打つ」運用とする
