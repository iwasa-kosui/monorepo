## ログにPIIが混入するとなぜ致命的か

サーバサイド開発において、ログにPII（個人識別情報）が混入する事故は珍しくありません。ユーザーのメールアドレスや氏名がログに書き込まれ、それがDatadogやNew Relicなどの外部SaaSに送信される。多くの開発チームにとっては「気をつけましょう」で済む問題かもしれません。

しかし、医療システムや金融システムのように厳格なデータ保護要件を持つドメインでは話が変わります。個人情報保護法やGDPR、HIPAAでは個人データの取り扱いに厳しい制約があり、ログ基盤として国外にデータセンターを持つSaaSを利用している場合、ログへのPII混入は個人データの国外移転に該当しうる。コンプライアンス違反であり、単なるバグではなくインシデントです。

厄介なのは、QAプロセスでログの中身が見落とされやすいことです。品質管理ではUIの使いやすさやAPIレスポンスの正しさに意識が向きがちで、「ロガーに何が渡されているか」を毎リリースで検証するチームは少ないのではないでしょうか。リリースのたびに全ログ出力をチェックするのは現実的ではないし、人手のチェックリストではスケールしません。

だからこそ仕組みで防ぐ必要があるわけですが、[前回の記事](/posts/2026/03/06/typescript-runtime-safety)で構造的部分型の話を書いたとき、「これログの文脈でも同じ問題が起きるな」と気づきました。今回はその話をします。

## DTOを作ったのにPIIが漏れるまで

「ドメインオブジェクトをそのままロガーに渡さなければいい。ログ用のDTOを分離すればいい」。これは正しい設計原則です。ただ、TypeScriptではこの原則を守ったつもりでもPIIが漏れます。

以下のコードを見てください。

```typescript
// ドメインオブジェクト: メールアドレスを含む
type User = {
  id: string;
  name: string;
  email: string; // PII
};

// ログ用DTO: PIIを含まないはず
type LogSafeUser = {
  id: string;
  name: string;
};

function createLogData(user: User): LogSafeUser {
  // スプレッド構文でDTOを作る。ありがちなパターン
  return { ...user };
}
```

`createLogData`の戻り値の型は`LogSafeUser`です。型検査は通ります。しかしランタイムでは、`{ ...user }`によって`email`フィールドもコピーされている。`JSON.stringify`すれば`email`が出力されます。

```typescript
const user: User = { id: "1", name: "Alice", email: "alice@example.com" };
const logData = createLogData(user);

console.log(JSON.stringify(logData));
// {"id":"1","name":"Alice","email":"alice@example.com"}
// ← 型上は LogSafeUser なのに、email が含まれている
```

なぜ型エラーにならないのか。前回の記事でも触れましたが、TypeScriptの構造的部分型では、`{ id, name, email }`は`{ id, name }`のスーパーセットであり、`LogSafeUser`として有効です。余剰プロパティチェックはオブジェクトリテラルの直接代入時のみ発動し、スプレッド構文や変数経由の代入では[チェックされません](https://www.typescriptlang.org/docs/handbook/type-compatibility.html)。[TypeScript Issue #19775](https://github.com/microsoft/TypeScript/issues/19775)でも仕様として確認されています。

つまり、DTO分離という設計原則を守り、型アノテーションも正しく付けたにもかかわらず、スプレッド構文ひとつでPIIが漏れる。型が「安全です」と言っているのに、ランタイムでは安全ではないわけです。

## 「型で守れるはず」という期待とその限界

DTO分離が破られるなら、もっと強力な型レベルの防御はないのか。いくつか検討してみます。

### Branded Typeでセンシティブなフィールドをマークする

```typescript
type Sensitive<T> = T & { readonly __sensitive: unique symbol };

type User = {
  id: string;
  name: string;
  email: Sensitive<string>; // センシティブとマーク
};
```

Branded Typeは型レベルの「タグ」であり、`Sensitive<string>`と`string`を区別できます。ロガーの引数型で`Sensitive`を含む型を拒否するConditional Typeを書けば、コンパイル時に検出できるかもしれません。

ただ、この防御は以下の場面で破綻します。

- スプレッド後にブランドが伝播するかは型定義次第で、フィールドを個別に取り出して再構築すれば消える
- `as`キャストで簡単にバイパスできる。`user.email as string`でブランドは消える
- そもそもランタイムには存在しない。Branded Typeはコンパイル時の概念であり、`JSON.stringify`の結果には何の影響も与えない

### Conditional Typesでロガーを型安全にする

```typescript
type ContainsSensitive<T> = T extends Sensitive<any>
  ? true
  : T extends object
    ? { [K in keyof T]: ContainsSensitive<T[K]> }[keyof T] extends false
      ? false
      : true
    : false;

type SafeLog<T> = ContainsSensitive<T> extends true ? never : T;

function log<T>(data: SafeLog<T>): void {
  logger.info(data);
}
```

一見エレガントですが、前節のDTO分離と同じ問題に帰着します。スプレッド構文で余剰プロパティを持つオブジェクトが`LogSafeUser`型として渡されたとき、型情報上は`Sensitive`なフィールドが存在しません。Conditional Typesはランタイムの実際のオブジェクト構造ではなく、TypeScriptが認識している型情報に基づいて判定するので、型が「emailは存在しない」と言っている以上、「安全」と判定してしまいます。

### 結局、TypeScript自体がSound型システムではない

これらの限界は個別のテクニックの問題ではなく、TypeScriptの設計思想に起因しています。[TypeScript公式FAQ](https://github.com/microsoft/TypeScript/wiki/FAQ)にもある通り、TypeScriptは意図的に完全なSound型システムではありません。型検査が通ったコードがランタイムで型通りに動く保証はない。

前回の記事でも書きましたが、型だけで全てを守れるという期待は、TypeScriptの設計思想と矛盾しています。

## ランタイム検証を主防御に据える

型に限界があるなら、ランタイムで実際にデータを除去・マスキングする仕組みを主防御に据えるべきです。ここでは2つのアプローチを紹介します。

### Zodのstripでログ出力前に余剰プロパティを除去する

[Zod](https://zod.dev/)の`z.object()`はデフォルトで余剰プロパティをstripします。ログに渡す前にスキーマを通せば、定義したフィールドのみが残ります。

```typescript
import { z } from "zod";

// ログに書き込んでよいフィールドだけを定義
const logSafeUserSchema = z.object({
  id: z.string(),
  name: z.string(),
});

type LogSafeUser = z.output<typeof logSafeUserSchema>;

function createLogData(user: User): LogSafeUser {
  // スキーマを通すことで、emailは除去される
  return logSafeUserSchema.parse(user);
}

const user: User = { id: "1", name: "Alice", email: "alice@example.com" };
console.log(JSON.stringify(createLogData(user)));
// {"id":"1","name":"Alice"}
// ← emailは含まれない
```

これはホワイトリスト方式です。「何を許可するか」を明示的に定義し、それ以外は全て除去する。ログに含めてよいフィールドを自分で選ぶため、意図しないPII混入を防げます。

スキーマ定義のコストはかかりますが、DTO手書き+コードレビューによる運用と比べて決定的に違うのは、Zodは実際にランタイムでデータを除去するという点です。型アノテーションは嘘をつけるが、`parse`の結果は嘘をつかない。前回の記事で「Zodの`z.object()`がGoのstructに相当する役割を担っている」と書きましたが、ログ出力の文脈でもまさに同じ話です。

### Pinoのredactでセンシティブフィールドを自動マスキングする

[Pino](https://github.com/pinojs/pino)のredactオプションは、指定したパスのフィールドをログ出力時に自動的にマスキングしてくれます。

```typescript
import pino from "pino";

const logger = pino({
  redact: {
    paths: ["email", "*.email", "user.email", "password", "*.password"],
    censor: "[REDACTED]",
  },
});

logger.info({ user: { id: "1", name: "Alice", email: "alice@example.com" } });
// {"user":{"id":"1","name":"Alice","email":"[REDACTED]"}}
```

こちらはブラックリスト方式です。「何を隠すか」を指定し、該当するフィールドをマスキングする。既存のコードベースに後付けで導入しやすいのが利点ですが、新しいセンシティブフィールドが追加されたときにredactリストの更新を忘れるリスクはあります。

### ホワイトリストとブラックリストの組み合わせ

理想的には両方を使います。

- Zod stripがホワイトリストとして機能し、ログ出力関数の入り口で許可されたフィールドだけを通す。これが主防御
- Pino redactがブラックリストとして機能し、Zodのstripをすり抜けた場合に既知のセンシティブフィールド名をキャッチする安全網になる

単一の防御に依存するのではなく、異なるアプローチを重ねることで、どちらか一方が破られても被害を最小化できます。

## 「忘れにくい設計」: 型とランタイムの正しい組み合わせ方

ここまでで「ランタイム検証が主防御」と述べました。ただ、ランタイム検証には致命的な弱点があります。Zodの`parse`を呼び忘れたら、何も防げません。

ここで型の出番です。型は「全てを防ぐ壁」ではなく、「ランタイム検証を忘れさせない仕組み」として使います。

```typescript
import { z } from "zod";

// ログに書き込んでよいフィールドのスキーマ
const logSafeUserSchema = z.object({
  id: z.string(),
  name: z.string(),
});

// Zodスキーマの出力型 = ログ関数が受け付ける型
type LogSafeUser = z.output<typeof logSafeUserSchema>;

// ロガーのラッパー: LogSafeUser しか受け付けない
function logUser(data: LogSafeUser): void {
  logger.info(data);
}

// --- 使う側 ---

const user: User = { id: "1", name: "Alice", email: "alice@example.com" };

// NG: User を直接渡すと型エラー（emailが余剰）
// ...ただし変数経由では型エラーにならない場合がある

// OK: Zodを通してから渡す
logUser(logSafeUserSchema.parse(user));
```

この設計のポイントは、ログ出力関数の引数型を`z.output<typeof schema>`にしていることです。スキーマの定義と型が自動的に同期するため、「型は更新したがスキーマは更新していない」という乖離が起きません。

ただし、前述の通り構造的部分型のため、変数経由で`User`オブジェクトを渡すと型エラーにならない場合があります。ここが型の限界です。だからこそ、この型制約はあくまで「ミスに気づきやすくする補助線」であり、実際にデータを除去するのはZodの`parse`です。

型を過信するのではなく、型の限界を理解した上で、型とランタイムにそれぞれ正しい役割を割り当てる。前回の記事から一貫して言っていることですが、この考え方はPII漏洩防止の文脈でも変わりません。

## まとめ

TypeScriptの型システムだけでログへのPII漏洩を防ぐことはできません。構造的部分型は余剰プロパティを透過させ、Branded TypeやConditional Typesもランタイムには無力です。これはTypeScriptの欠陥ではなく設計思想です。

型の限界を認めた上で、以下の多層防御を設計に組み込むのがよいと考えています。

1. ログ出力の境界にZod stripを挟む。ログに書き込んでよいフィールドをスキーマで定義し、`parse`を通すことで余剰プロパティを実際に除去する。これが主防御
2. Pino redactをフォールバックとして設定する。既知のセンシティブフィールド名をredactリストに登録し、Zodをすり抜けた場合のセーフティネットにする
3. ログ出力関数の引数型をZodスキーマの出力型に制約する。`z.output<typeof schema>`を使い、スキーマと型の乖離を防ぐ。型は「全てを防ぐ壁」ではなく「parseの呼び忘れに気づきやすくする仕組み」として使う

「型安全な開発」という言葉に安心せず、型にできることとできないことを正確に理解すること。[前回の記事](/posts/2026/03/06/typescript-runtime-safety)と合わせて読んでもらえれば、サーバサイドTypeScriptで型とランタイムをどう組み合わせるべきかの全体像が見えてくるのではないかと思います。
