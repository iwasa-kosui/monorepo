## SDDの「仕様」とは何を指しているのか

仕様駆動開発（Spec-Driven Development, SDD）が話題だ。[Kiro](https://kiro.dev/)、[OpenSpec](https://openspec.dev/)、[Spec Kit](https://github.blog/ai-and-ml/generative-ai/spec-driven-development-with-ai-get-started-with-a-new-open-source-toolkit/)といったツールが登場し、[Martin Fowlerのブログ](https://martinfowler.com/articles/exploring-gen-ai/sdd-3-tools.html)でも詳細に分析されている。一方で、SDDの是非を巡る議論は空中戦になりがちだ。その原因は「仕様」という言葉が指す範囲が人によって異なるからだと考えている。

各ツールの定義を確認してみよう。

Martin Fowler（Birgitta Böckeler）の記事では、specを次のように定義している。

> A structured, behavior-oriented artifact - or a set of related artifacts - written in natural language that expresses **software functionality** and serves as guidance to AI coding agents.

Kiroのspecは `requirements.md`（EARS記法によるユーザーストーリーと受け入れ基準）→ `design.md`（アーキテクチャと技術判断）→ `tasks.md`（実装タスク）の3ファイルで構成される。「なぜこのシステムを作るのか」に相当する独立した成果物はなく、ユーザーの自然言語プロンプトがその役割を担う。

OpenSpecは少し異なり、`proposal.md`（"why we're doing this, what's changing"）→ `specs/`（振る舞い契約）→ `design.md` → `tasks.md` の4ファイル構成だ。Whyを扱うproposal層が存在する点でKiroより上流をカバーしている。

ただし、OpenSpecの `proposal.md` が扱う「Why」は「なぜこの変更をするか」という機能変更レベルのWhyだ。「なぜこのシステムを作るのか」「セールスオペレーションチームはCRMに対してどのタイミングでどんな入出力を行うのか」「各運用チームの業務目標は何か」——こうした事業・業務レベルのWhyを構造的に分析するフレームワークではない。

ソフトウェア開発には「要望→要求→要件→仕様→設計」という階層がある（[ゴトーラボの解説](https://www.gotolab.co.jp/%E8%A6%81%E6%9C%9B-%E8%A6%81%E6%B1%82-%E8%A6%81%E4%BB%B6-%E4%BB%95%E6%A7%98-%E8%A8%AD%E8%A8%88%E3%81%AE%E9%81%95%E3%81%84%E3%81%AB%E3%81%A4%E3%81%84%E3%81%A6%E8%A7%A3%E8%AA%AC/)が分かりやすい）。SDDが主に扱うのは「要件→仕様→設計」のレイヤーだ。「要望→要求」の構造化——すなわち、ステークホルダーの業務を理解し、システム化の目的とスコープを明確にする工程——はSDDの前工程であり、両者は補完関係にある。

本記事では、この前工程を[RDRA](https://www.rdra.jp/)とClaude Codeで実践するワークフローを紹介する。

## 要求分析が抜けるとどうなるか

N=1の経験だが、構造的な問題として共有する。

セールスオペレーションやカスタマーサクセスなど、顧客向き合いの運用チームが使うシステムを開発する場面を考えてほしい。例えばCRMへのデータ書き込み機能を開発するとき、以下のような判断が求められる。

- 書き込みは一括で行うべきか、個別に行うべきか
- 書き込みが失敗した場合、どのようなメッセージを誰に通知すべきか
- どのタイミングで、どのチームのメンバーがこの機能を使うのか

これらの判断は、それぞれの運用チームが何を目標とし、何に関心を置いて業務をしているかによって大きく変わる。にもかかわらず、曖昧な仮定を置いたまま開発を進めてしまうと、運用コストが増大する。そして多くの場合、そのまま運用が開始され、コストが高いシステムを使い続けることになる。本来そのチームが向き合いたかった目標を達成するための時間が奪われ、最終的にはいびつな運用プロセスが固着してしまう。

SDDで仕様を丁寧に書いたとしても、その仕様の前提となる業務理解が間違っていれば意味がない。仕様の前に要求がある。

## なぜRDRAか——表形式がコーディングエージェントに向いている

要求分析のフレームワークはRDRAだけではない。イベントストーミング、ユーザーストーリーマッピング、ICONIX、匠Methodなど、選択肢は多い。その中でRDRAを選ぶ理由は、**コーディングエージェントとの相性**にある。

RDRAの特徴は、要求を4つのレイヤー（システム価値/外部環境/システム境界/システム）で構造化し、各要素を**表形式**で表現する点だ。図も使うが、RDRAの成果物の本体は表——つまり構造化されたテキストデータだ。

これがなぜ重要か。イベントストーミングを例に考えてみよう。イベントストーミングはFigJamやMiroで付箋を空間的に配置するビジュアルな手法だ。ワークショップとしては非常に強力だが、その結果をコーディングエージェントに渡すのが難しい。付箋の位置関係、グルーピングの境界、矢印の意味——これらの空間的な情報をテキストに構造化する段階で、情報が欠落しやすい。セクション境界をエージェントがうまく読み取ってくれないことが経験上多い。

一方RDRAの表形式であれば、YAMLフロントマター付きのMarkdownでそのまま構造化データとして表現できる。実際に私が使っているClaude Codeのspecスキルでは、RDRAの成果物を以下のようなフォーマットで管理している。

```yaml
---
type: rdra-context
id: "BIZ-001"
name: "contract-management"
display_name: "契約管理"

value:
  goals: ["GOAL-001"]
  requirements:
    - id: "REQ-001"
      description: "契約ステータスの変更をリアルタイムに把握できる"
      traces_to: ["GOAL-001"]

environment:
  business_usecases:
    - id: "BUC-001"
      name: "新規契約登録"
      actors: ["ACTOR-001"]
      traces_to: ["REQ-001"]

boundary:
  usecases:
    - id: "UC-001"
      name: "契約情報を入力する"
      actors: ["ACTOR-001"]
      traces_to: ["BUC-001"]
---
```

`traces_to` による単方向の参照で、GOAL → REQ → BUC → UC → FR → SPEC → TASK というWhyの依存チェーンが形成される。「このタスクはなぜ必要か」を遡れば必ずゴールに到達する。エージェントにとっても人間にとっても、構造が明示的で扱いやすい。

## Claude Code × RDRAの実践ワークフロー

ここからが本題だ。私が実際に業務で行っているワークフローを紹介する。

### Step 1: スコープ把握——エージェントが社内ドキュメントを探索する

プロジェクトを推進する数名のエンジニアが、まずインセプションデッキやOKR、関連チームの業務マニュアルを読む。その上で、Claude Codeのエージェントに社内のConfluenceやNotion、Slackを検索させ、関連するアクターや外部システムの仮説を立てさせる。

エージェントの出力は完璧ではない。だが、仮説があることで対話が始まる。エンジニアは「いや、このチームも関わっている」「過去のアーキテクチャ俯瞰図を見ると、このシステムも関係しそうだ」とフィードバックを返す。エージェントはフィードバックを反映し、RDRA形式の `overview.md`（アクター一覧・ゴール一覧・コンテキスト一覧）を生成する。

### Step 2: 業務フロー生成——as-isの仮説検証

コンテキスト（業務領域）が特定できたら、エージェントに各コンテキストのas-is業務フローをMermaidシーケンス図で生成させる。同時に、各業務プロセスでどんな課題があるかの仮説も提示させる。

ここで人間のレビューが効く。例えば「エアコンのプロビジョニングと店舗のプロビジョニングは実際には別の業務フローだから、分けて書いてほしい」というフィードバックが出る。こういった業務の実態は、ドキュメントには書かれていないことが多い。エンジニアの頭の中にある暗黙知がエージェントの仮説によって引き出される。

as-is業務フローが固まったら、to-be候補を複数提案させる。各候補がどのゴール・要求を満たすかを `traces_to` で紐づけることで、「なぜこの改善案を選ぶのか」の判断材料が構造化される。

### Step 3: 非同期の仮説検証ループ

このワークフローの隠れた価値は、**エージェントの処理待ち時間が人間の思考時間になる**ことだ。

エージェントが業務フローを生成したり、to-be候補を検討している間に、人間は別のドキュメントを読み込んだり、チーム内で議論を再開したりできる。エージェントの出力を待ってから次に進むのではなく、人間とエージェントが非同期に仮説を検証し合うリズムが生まれる。

### Step 4: 成果物の共有——GitHubの外で育てる

要求分析の成果物は、最終的にセールスオペレーションやカスタマーサクセスといった運用チームにレビューしてもらう必要がある。ここで「成果物をどこに置くか」が問題になる。

GitHubで管理するのが常にベストとは限らない。理由は2つある。

**第一に、コストとリテラシーの壁がある。** GitHubのライセンスは1シートあたりのコストが馬鹿にならない。非エンジニアを含む運用チーム全員にアカウントを発行するのはコスト面で厳しいし、GitHubを使いこなすリテラシーを全員に求めるのは組織をスケールする上でボトルネックになる。

**第二に、リアルタイム共同編集の価値がある。** ConfluenceやNotionであれば、ミーティング中に全員が同時に要求を修正できる。ローカルにあるMarkdownファイルをGitHubにプッシュするフローだと、誰か一人が議事録を取るついでに修正しなければならなくなる。全員が話しながらドキュメントを育て、その結果をClaude Codeに読み込ませて次のイテレーションに反映する——このサイクルが回ることで、要求分析がエンジニアだけの作業ではなくチーム全体の営みになる。

もちろん、スタートアップのアーリーフェーズでリテラシーの高いメンバーが揃っているなら、GitHubで一元管理するのも合理的だ。チームの実態に合わせて選べばよい。

## まとめ

SDDと要求分析は対立しない。SDDは「仕様を書いてからコードを生成する」アプローチであり、その前工程として「なぜこのシステムを作るのか、誰のどんな業務課題を解決するのか」を構造化する要求分析がある。両者は補完関係にある。

RDRAの表形式（YAML + Markdown）はコーディングエージェントとの相性が良い。FigJamやMiroのビジュアル手法では構造化しにくい情報を、エージェントが読み書きしやすいフォーマットで表現できる。`traces_to` によるトレーサビリティは、「なぜこのタスクが必要なのか」をゴールまで一気通貫で遡れる仕組みを提供する。

そして成果物の配置先はチームの実態に合わせる。GitHubが全てではない。非エンジニアが気軽に書き込み、議論しながら育てられるツールと連携することで、要求分析は初めてチーム全体のものになる。
