## はじめに — なぜアーキテクチャパターンの選定が必要だったのか

私が所属するチームは、医療SaaS企業の認証・ID管理・ライセンス管理などの共通基盤を担当しています。社内の複数プロダクトが依存する基盤であり、この基盤が壊れると全プロダクトに影響が出ます。

基盤には認証基盤、ID基盤、ライセンス基盤、端末・証明書基盤の4つのシステムがあり、それぞれ異なる品質要求を持っています。

### 品質要求が相反する問題

例えば認証基盤とID基盤の関係を見てみましょう。

- **認証基盤**: 可用性が最優先。ログインできなければ医療現場が止まる
- **ID基盤**: 整合性とトレーサビリティが最優先。患者データの真正性を保証する必要がある

認証基盤はID基盤に依存しています。この依存関係が具体的にどう困るかというと、

- **メンテナンス計画の衝突**: ID基盤で整合性維持のために停止してデータ移行したいが、認証基盤は停止できない
- **障害の連鎖**: ID基盤でデータ不整合が発生した場合、認証基盤も巻き込まれて停止せざるを得ない

品質要求が異なるシステムが密結合していることで、一方の改善が他方に悪影響を及ぼす状況でした。

### チームの制約

さらにチームは非常に小規模で、Embedded SREを迎える余裕はありませんでした。開発チーム自身が信頼性に責任を持つ必要があり、運用負荷の低いアーキテクチャを選ぶことが重要でした。

この状況で、品質要求が異なるシステムの**独立性を高めたい**というのが出発点です。

## アーキテクチャパターンの比較

求められる要件を整理すると、以下の3点でした。

1. 基盤システムは**発展途上**であり、誤った分割は障害点を増やすリスクがある
2. データの**強い整合性・一貫性**を保ちたい
3. サービスごとの**独立したデプロイ**を実現したい

これらの要件を踏まえて、4つの代表的なアーキテクチャパターンを比較検討しました。

### マイクロサービス — 不採用

サービスごとに独立したDBとデプロイを持ち、API通信で連携するパターンです。

**強み**: サービス単位の独立デプロイと柔軟なスケーリング

**懸念**: 基盤がまだ発展途上のため、誤った境界で分割してしまうリスクが大きく、分散トランザクションの複雑さも課題でした。小規模チームで分散システムの運用負荷を抱えるのは現実的ではないと判断しました。

### イベント駆動アーキテクチャ — 不採用

ProducerがEvent Brokerにイベントを発行し、Consumerが非同期に処理するパターンです。

**強み**: 疎結合・非同期処理による高いスケーラビリティ

**懸念**: 結果整合性が前提となるため、私たちが求める強いデータ整合性を保証できません。またEvent Brokerが新たな障害点となり、小規模チームでの運用負荷が懸念でした。

### モジュラモノリス — 不採用

単一デプロイユニットの中でモジュールをインターフェースで分離し、単一データベースで強い整合性を保つパターンです。

**強み**: 単一DBで強い整合性を保ちつつモジュール分離が可能

**懸念**: デプロイが一体のため、認証基盤だけを緊急デプロイするといった対応が困難でした。例えば脆弱性対策のパッチを認証基盤だけに適用したいケースに対応できません。

### サービスベースアーキテクチャ — 採用

**上記3つのいずれも要件を完全には満たせない**中で、私たちが採用したのがサービスベースアーキテクチャです。

## サービスベースアーキテクチャとは

サービスベースアーキテクチャは、Mark Richards, Neal Fordの『[ソフトウェアアーキテクチャの基礎](https://www.oreilly.co.jp/books/9784873119823/)』で体系化されたパターンです。

特徴は以下の通りです。

- 単一の**共有DB**を持ちながら、サービスは**独立してデプロイ**できる
- **サービス間のAPI通信は禁止**
- サービス間で**DBの読み取りを許可**
- ただし各テーブルは**特定のサービスが所有**し、他のサービスによる書き込みは禁止

マイクロサービスとモノリスのいいとこ取りと言えます。分散トランザクションの複雑さを避けつつ、デプロイの独立性を確保できます。

### 「共有DBってアンチパターンでは?」

この疑問はもっともです。マイクロサービスの文脈では「各サービスは自分専用のDBを持つべき」と言われますし、共有DBは密結合の原因になると警告されています。

しかし、サービスベースアーキテクチャは「共有DBアンチパターン」とは明確に異なります。

|  | 共有DBアンチパターン | サービスベースアーキテクチャ |
|---|---|---|
| テーブルへのアクセス | どのサービスもどのテーブルも自由に読み書き | テーブルごとに**所有権**を持つ |
| 書き込み権限 | 制限なし | **所有サービスのみ**書き込み可能 |
| 他サービスのデータ | 自由に変更可能 | **読み取りのみ** |
| 変更の影響範囲 | 不明 | 所有権により**追跡可能** |

ポイントは所有権の概念です。各テーブル（またはスキーマ）には明確な所有者がおり、所有者だけが書き込みを行います。他のサービスは読み取りのみです。これにより、変更の影響範囲を把握でき、密結合を防げます。

### なぜAPI通信ではなくDB直接参照なのか

```typescript
// サービス間API呼び出し
const user = await userService.getUser(userId);

// 共有DBから直接読み取り
const user = await db.query(
  'SELECT * FROM directory.users WHERE id = $1',
  [userId]
);
```

API呼び出しにはDNS遅延やネットワーク障害による連鎖障害のリスクがあります。私たちも実際にそうした問題を経験しました。

DB直接参照のメリットは以下の通りです。

- ネットワーク障害・DNS遅延の影響を受けない
- トランザクションで整合性を保証できる
- 低レイテンシー

私たちの目的は「独立デプロイ」であって「ネットワーク分離」ではありません。この点が重要です。

## 実装 — PostgreSQLでの具体的な構成

### スキーマ分割

単一のPostgreSQL内に、サービスごとのスキーマを作成します。

```
PostgreSQL
├── auth スキーマ        ← 認証サービスが所有
├── directory スキーマ   ← ディレクトリサービスが所有
└── asset スキーマ       ← 端末管理サービスが所有
```

物理的には1つのDBですが、論理的にスキーマで分離しています。

### DBユーザーによる権限制御

「自分のスキーマのみ書き込み、他は読み取りのみ」という原則を、運用ルールではなく**DBの権限として強制**します。

```sql
-- 認証サービス用のロール
CREATE ROLE auth_service;
GRANT ALL ON auth.* TO auth_service;
GRANT SELECT ON directory.* TO auth_service;

-- ディレクトリサービス用のロール
CREATE ROLE directory_service;
GRANT ALL ON directory.* TO directory_service;
GRANT SELECT ON auth.*, asset.* TO directory_service;
```

認証サービス（`auth_service`）はauthスキーマにフルアクセスできますが、directoryスキーマはSELECTのみです。ディレクトリサービス（`directory_service`）も同様に、自分のスキーマだけ書き込み可能です。

これはRLS（Row Level Security）と同じ「仕組みで防ぐ」思想です。開発者が間違えても、DBレベルでブロックされます。

### RLSとの組み合わせ

サービスベースアーキテクチャのスキーマ所有権とRLSは、異なるレイヤーの分離を担っています。

- **スキーマ所有権**: サービス間の分離（水平方向）
- **RLS**: テナント間の分離（垂直方向）

これらを組み合わせることで、「認証サービスがディレクトリサービスのデータを書き換える」ことも、「テナントAがテナントBのデータを参照する」ことも、仕組みとして不可能になります。

## 発展可能性 — 将来のアーキテクチャ進化

サービスベースアーキテクチャの大きな利点は発展可能性です。

- **サービスベース → マイクロサービス**: ドメイン境界が明確になった段階で、特定のサービスをDBごと分離できる
- **サービスベース → イベント駆動の部分導入**: 非同期処理が必要になった箇所だけ、イベント駆動パターンを導入できる
- **逆方向（マイクロサービス → 再統合）は困難**: 一度完全に分離すると、再統合は大規模な作り直しになる

つまり、サービスベースアーキテクチャは「分割の選択肢を残しつつ、今は強い整合性を維持する」ポジションです。

早すぎる分離は間違った境界での分割を招きます。「今わからないことを認め、将来の選択肢を残す」設計判断が、サービスベースアーキテクチャの本質だと考えています。

## 課題と教訓 — 銀の弾丸ではない

もちろん課題もあります。

### 参照可否の区別

サービスが拡大すると、テーブルの区別が曖昧になります。

- 他サービスから参照して**良い**テーブル（公開テーブル）
- 他サービスから参照**してほしくない**テーブル（内部テーブル）

私たちは命名規則やスキーマ分割で公開/非公開を明示し、チーム間でルールを明文化して対応しています。公開テーブルは事実上のAPIインターフェースであり、安易に変更できないことを認識する必要があります。

### スキーマ変更の制約

他サービスが参照するテーブルのカラムを変更するには、影響範囲の調査とデプロイ順序の調整が必要です。

例えば、directoryスキーマのusersテーブルにカラムを追加する場合、authサービスがそのテーブルを参照していれば、authサービスのコードも確認が必要です。カラム削除やリネームの場合は、参照元サービスの更新を先にデプロイする必要があります。

公開テーブルは慎重に設計するようにしていますが、デプロイの柔軟性については正直まだ課題が残っています。

### 運用から得た気づき

- **所有権の明文化が重要**: 「このテーブルは誰が所有しているか」を文書化し、全メンバーがアクセスできる場所に置くことが重要です。ADR（Architecture Decision Records）でテーブルの所有権変更も記録しています
- **SELECT権限の付与は慎重に**: 一度他サービスに読み取り権限を与えると、依存関係が生まれます。本当にDB直接参照が必要か、データ基盤経由で十分ではないかを都度検討しています
- **テスト環境でも権限を再現する**: 開発環境でスーパーユーザーを使っていると、本番で初めて権限エラーに気づく事態が起きます。CI環境でも本番と同じロール設定を再現することが重要です

## まとめ — サービスベースアーキテクチャが適するケース

サービスベースアーキテクチャは、以下のようなケースで特に有効だと考えています。

- **データの強い整合性が必要**だが、デプロイは独立させたい
- **ドメイン境界がまだ不明確**で、早すぎる分割を避けたい
- **小規模チーム**で、分散システムの運用負荷を最小限に抑えたい
- 将来的にマイクロサービスやイベント駆動への**段階的な進化**を見据えたい

銀の弾丸ではありませんが、「マイクロサービスかモノリスか」という二項対立に陥りがちなアーキテクチャ選定において、検討に値する選択肢です。

設計パターンは「選ぶ」ものではなく「育てる」もの。選んだ後も運用しながら課題を見つけ、設計にフィードバックし続けることが本質だと、私たちは実践を通じて学びました。
