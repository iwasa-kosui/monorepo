## 品質要求が相反する問題

私たちのチームは認証基盤、ID基盤、ライセンス基盤、端末・証明書基盤の4つのシステムを担当しています。社内の複数プロダクトが依存する共通基盤で、ここが壊れると全プロダクトに影響が出ます。

![プラットフォームの全体構成。プロダクト群がOIDCやAPIでプラットフォーム内の認証基盤・ライセンス基盤・端末/証明書基盤・ID基盤を参照し、ID基盤がデータ基盤へ反映する](/images/service-based-architecture/platform-overview.svg)

厄介なのは、これらのシステムがそれぞれ異なる品質要求を持っていることです。

認証基盤は「ログインできなければ医療現場が止まる」ので可用性に全振り。一方、ID基盤は患者データの真正性を保証する必要があるから整合性とトレーサビリティを譲れない。そして認証基盤はID基盤に依存している。

![認証基盤とID基盤の依存関係。認証基盤がID基盤を参照しており、この依存が品質要求の相反を引き起こす](/images/service-based-architecture/auth-id-dependency.svg)

この依存関係が何を引き起こすかというと、

- ID基盤で整合性維持のために停止してデータ移行したいのに、認証基盤は止められない
- ID基盤でデータ不整合が起きると、認証基盤も巻き込まれて停止せざるを得ない

品質要求が異なるシステムが密結合していて、一方を改善しようとすると他方に悪影響が出る。この状況をどうにかする必要がありました。

加えて、チームは非常に小規模で、アプリケーション開発者の採用は進んだもののEmbedded SREを迎える余裕はありませんでした。運用負荷の低いアーキテクチャを選ぶしかない、という制約もあります。

## 何を選び、何を選ばなかったか

3つの要件が衝突していました。

1. 基盤はまだ発展途上で、下手に分割すると障害点が増える
2. でもデータの整合性は絶対に譲れない
3. だけどシステムごとに独立デプロイしたい

この3つを同時に満たすアーキテクチャパターンを探して、4つの候補を検討しました。

### マイクロサービス

![マイクロサービスアーキテクチャ。サービスごとに独立したDB・デプロイ・API通信を持つ構成](/images/service-based-architecture/arch-microservices.svg)

最初に検討したのはマイクロサービスです。独立デプロイできるし、スケーリングも自由。ただ、実際に設計を詰めていくと「どこで切るのか」が決まらない。4つのシステムは本来別の責務だけど、参照関係が絡み合っていて、誤った境界で引いてしまうと後が大変です。分散トランザクションの複雑さもあり、当時のチーム規模では現実的ではないと判断しました。

### イベント駆動アーキテクチャ

![イベント駆動アーキテクチャ。ProducerがEvent Brokerを介してConsumerに非同期でイベントを配信する構成](/images/service-based-architecture/arch-event-driven.svg)

疎結合でスケーラビリティが高い。ただ、結果整合性が前提になるので、私たちが求める強いデータ整合性とは相性が悪い。Event Brokerが新たな障害点になるし、イベント消失時のリカバリなど運用負荷が気になりました。

### モジュラモノリス

![モジュラモノリスアーキテクチャ。単一デプロイユニット内でモジュール分離しつつ、単一データベースで強い整合性を確保する構成](/images/service-based-architecture/arch-modular-monolith.svg)

単一DBで整合性を保ちつつモジュール分離ができる。整合性の要件は満たせますが、デプロイが一体なのが困る。認証基盤だけ脆弱性パッチを当てたい、みたいなケースに対応できません。

### サービスベースアーキテクチャ

上の3つのどれも要件を完全に満たせない中で、『[ソフトウェアアーキテクチャの基礎](https://www.oreilly.co.jp/books/9784873119823/)』（Mark Richards, Neal Ford著）を読んでいて「これだ」と思ったのがサービスベースアーキテクチャでした。

## サービスベースアーキテクチャとは

![サービスベースアーキテクチャ。各サービスが独立デプロイ可能で、単一PostgreSQL内のスキーマを所有。サービス間のAPI通信は禁止し、他スキーマへはSELECTのみ許可する構成](/images/service-based-architecture/service-based-architecture.svg)

ざっくり言うと、こういうパターンです。

- 単一のDBを共有しつつ、サービスは独立してデプロイできる
- サービス間のAPI通信は禁止
- サービス間でDBの読み取りは許可
- ただし、各テーブルは特定のサービスが所有し、他サービスは書き込めない

マイクロサービスの「独立デプロイ」と、モノリスの「単一DBによる強い整合性」を両取りする設計です。分散トランザクションの複雑さを避けつつ、デプロイの独立性を確保できます。

### 「それ、ただのDB共有では?」

「DB共有はアンチパターンでは?」という質問をよくもらいます。確かに、マイクロサービスの教科書には「各サービスは自分のDBを持て」と書いてある。

ただ、ここで言うDB共有アンチパターンは「どのサービスもどのテーブルも自由に読み書きできる状態」を指しています。サービスベースアーキテクチャは違います。

|  | DB共有アンチパターン | サービスベースアーキテクチャ |
|---|---|---|
| テーブルへのアクセス | どのサービスもどのテーブルも自由に読み書き | テーブルごとに所有権を持つ |
| 書き込み権限 | 制限なし | 所有サービスのみ書き込み可能 |
| 他サービスのデータ | 自由に変更可能 | 読み取りのみ |
| 変更の影響範囲 | 不明 | 所有権により追跡可能 |

キモは所有権です。各テーブル（またはスキーマ）には明確な所有者がいて、所有者だけが書き込む。他のサービスは読み取りのみ。これが「何でもありのDB共有」とは決定的に違う点です。

### なぜAPI通信ではなくDB直接参照なのか

```typescript
// サービス間API呼び出し
const user = await userService.getUser(userId);

// DB直接参照
const user = await db.query(
  'SELECT * FROM directory.users WHERE id = $1',
  [userId]
);
```

以前、サービス間をAPI経由でデータ参照していた時期がありましたが、DNSキャッシュの問題でレスポンスが不安定になった経験があります。API呼び出しはネットワーク障害による連鎖障害のリスクを常に抱えている。

DB直接参照なら、ネットワーク遅延の影響を受けないし、トランザクション内で整合性も保証できる。私たちの目的は「独立デプロイ」であって「ネットワーク分離」ではないので、DB直接参照で十分でした。

## PostgreSQLでの実装

### スキーマ分割

![PostgreSQL内のスキーマ分割。auth/directory/assetスキーマがそれぞれ認証/ID/ライセンスサービスに所有され、サービス間はSELECT参照のみ](/images/service-based-architecture/shared-db-schemas.svg)

単一のPostgreSQL内に、サービスごとのスキーマを作ります。

```
PostgreSQL
├── auth スキーマ        ← 認証サービスが所有
├── directory スキーマ   ← ディレクトリサービスが所有
└── asset スキーマ       ← 端末管理サービスが所有
```

物理的には1つのDBですが、論理的にスキーマで分離しています。

### DBユーザーによる権限制御

「自分のスキーマのみ書き込み、他は読み取りのみ」を運用ルールではなくDBの権限として強制します。ここが大事なポイントです。

```sql
-- 認証サービス用のロール
CREATE ROLE auth_service;
GRANT ALL ON auth.* TO auth_service;
GRANT SELECT ON directory.* TO auth_service;

-- ディレクトリサービス用のロール
CREATE ROLE directory_service;
GRANT ALL ON directory.* TO directory_service;
GRANT SELECT ON auth.*, asset.* TO directory_service;
```

`auth_service` はauthスキーマにフルアクセスできますが、directoryスキーマはSELECTのみ。ルールじゃなくて、DBレベルで制御している。開発者がうっかりINSERTしようとしてもブロックされます。

RLS（Row Level Security）でテナント間の分離を行っているのと同じ「仕組みで防ぐ」思想です。

### RLSとの組み合わせ

スキーマ所有権とRLSは、異なるレイヤーの分離を担っています。

- スキーマ所有権: サービス間の分離（水平方向）
- RLS: テナント間の分離（垂直方向）

この2つを組み合わせることで、「認証サービスがディレクトリサービスのデータを書き換える」ことも、「テナントAがテナントBのデータを参照する」ことも、仕組みとして不可能になります。

## 発展可能性

サービスベースアーキテクチャを選んだ一番の理由は、実は発展可能性にあります。

ドメイン境界が明確になったら、特定のサービスをDBごと分離してマイクロサービス化できる。非同期処理が必要になった箇所だけ、イベント駆動パターンを部分的に導入できる。逆に「まだ分割しない」という選択もできる。

一方で、一度マイクロサービスとして完全に分離してしまうと、再統合は大規模な作り直しになります。分離は不可逆に近い操作なんです。

早すぎる分離は間違った境界での分割を招く。「今わからないことを認めて、将来の選択肢を残す」というのが、このアーキテクチャの本質だと思っています。

## 課題

もちろん課題もあります。

### テーブルの公開/非公開が曖昧になる

運用して3ヶ月くらいで「このテーブル読んでいいんですか?」という質問が増えました。directoryスキーマのusersテーブルが、あるチームでは「認証のための読み取り専用テーブル」として、別のチームでは「従業員マスタの完全なソース」として参照されていて、利用パターンが分かれていた。

結局、命名規則で「このテーブルは公開、このテーブルは内部専用」と明示することにしました。公開テーブルは事実上のAPIインターフェースなので、安易にスキーマを変えられないことを認識しておく必要があります。

### スキーマ変更にはデプロイ順序の調整が必要

他サービスが参照しているテーブルのカラムを変更するには、影響範囲の調査とデプロイ順序の調整が必要です。カラム削除やリネームの場合は、参照元サービスの更新を先にデプロイしないといけない。正直、デプロイの柔軟性についてはまだ課題が残っています。

### 運用で学んだこと

- テーブルの所有権を明文化してADRに記録する。新しいメンバーが「このテーブルは誰のもの?」と迷わないようにする
- SELECT権限の付与は慎重に。一度読み取り権限を渡すと依存関係が生まれる。本当にDB直接参照が必要なのか、データ基盤経由で十分じゃないのか、都度検討する
- テスト環境でも本番と同じロール設定を再現する。開発環境でスーパーユーザーを使っていると、本番で初めて権限エラーに気づくことになる

## まとめ

サービスベースアーキテクチャは、データの強い整合性が必要で、でもデプロイは独立させたくて、ドメイン境界もまだ不明確で、チームも小さい、という状況にフィットするアーキテクチャです。

「マイクロサービスかモノリスか」という二項対立に陥りがちなアーキテクチャ選定ですが、その間にこういう選択肢もあるということを知ってもらえたらと思います。

設計パターンは選んで終わりではなくて、運用しながら課題を見つけて、フィードバックし続けるものだと実感しています。
