## はじめに

インシデント対応の後にやってくるポストモーテム。対応そのものも大変ですが、ポストモーテムの**準備**もまた大きな負担です。

Slack のインシデントチャンネル、Datadog のアラートやダッシュボード、Jira のチケット、GitHub の Issue や Pull Request、Confluence のランブック ── 情報はあちこちに散らばっていて、それを時系列に整理し、テンプレートに落とし込む作業は、インシデント直後の疲弊した状態ではとにかく辛いものです。

この記事では、[Claude Code](https://docs.anthropic.com/en/docs/claude-code) のカスタムスキル機能を使って、ポストモーテムの準備作業を大幅に軽くするための設計指針を紹介します。

なお、incident.io や Rootly などのインシデント管理プラットフォームは、すでに AI によるポストモーテム自動生成機能を提供しています。これらのツールを導入済みであれば、そちらを使うのが最も手軽です。この記事が対象とするのは、そうしたプラットフォームを導入していない、あるいは自チームの既存ワークフローに合わせたカスタマイズが必要なケースです。

## ポストモーテム準備の何が辛いのか

ポストモーテム準備の辛さは、大きく4つに整理できます。

### 1. 情報源が多すぎる

インシデントに関する情報は、単一のツールに閉じていません。

- **コミュニケーション**: Slack のインシデントチャンネル
- **モニタリング**: Datadog のアラート・ログ・APM
- **タスク管理**: Jira のチケット
- **コード**: GitHub の Issue・Pull Request
- **ドキュメント**: Confluence のランブック・過去のポストモーテム

これらを横断的に見ながら「何が起きたか」を再構成する必要があります。

### 2. テキスト量が膨大

Slack のインシデントチャンネルには大量のメッセージが流れます。Datadog のアラートも複数発火しているでしょう。これらすべてに目を通し、関連する情報を拾い出すのは骨が折れます。

### 3. タイムライン作成が手間

「いつ、何が起きたか」の時系列を正確に組み立てるには、複数の情報源のタイムスタンプを突き合わせる必要があります。Slack のメッセージ、アラートの発火時刻、デプロイのタイムスタンプ ── これらを手作業で時系列に並べ替えるのは、単純だけれど時間のかかる作業です。

### 4. インシデント直後は疲弊している

そもそも、この準備作業はインシデント対応が終わった直後にやることが多いです。長時間の対応で疲弊した状態で、冷静に情報を整理するのは精神的にも体力的にもしんどいものです。

## なぜコーディングエージェントがポストモーテム準備に向いているのか

ここで少し視点を広げて、コーディングエージェントの本質的な価値について考えてみます。

コーディングエージェントの強みは、コードを書くことだけにあるわけではありません。**分断された様々なデータソースにある情報を収集し、統合できること** ── これこそがコーディングエージェントの本質的な価値だと考えています。

CLI を叩いて GitHub の PR を取得し、MCP サーバー経由で Slack のログを引っ張り、Datadog の API からメトリクスを収集する。これらを一つのコンテキストに統合して、人間が読める形に構造化する。ポストモーテム準備に必要な作業は、まさにこの「分断された情報の収集と統合」です。

実際、Google Cloud の SRE チームは Gemini CLI を使ってインシデントの会話履歴、メトリクス、ログからポストモーテムを自動生成しています（[InfoQ, Feb 2026](https://www.infoq.com/news/2026/02/google-sre-gemini-cli-outage/)）。Datadog の Bits AI SRE は、アラート発火時に自律的にランブックやテレメトリを分析し、オンコール担当者がログインする前に結論を出します（[Datadog Blog, Dec 2025](https://www.datadoghq.com/blog/bits-ai-sre/)）。

### スキル設計なしでは一般論をなぞるだけで終わる

ただし、エージェントに「ポストモーテムを書いて」とだけ伝えても、うまくはいきません。

スキルをきちんと設計しないと、2つの問題が起きます。

**問題1: 表層的な一般論で終わる。** 具体的なデータソースや取得手順を指示しなければ、エージェントは「タイムラインを整理しましょう」「影響範囲を特定しましょう」といった一般論を並べるだけで、実際のインシデントデータに基づいたドラフトは生成されません。

**問題2: コンテキストが爆発して精度が落ちる。** Slack の全メッセージ、Datadog の全アラート、GitHub の全 PR を一度にコンテキストに詰め込もうとすると、コンテキストウィンドウが溢れ、情報の取捨選択が雑になり、出力の品質が大きく劣化します。研究では、関連情報がコンテキストの中間部分に埋もれると正確性が30%以上低下する "Lost in the Middle" 問題が報告されています。

これが、Anthropic が「プロンプトエンジニアリング」ではなく「**コンテキストエンジニアリング**」── コンテキストウィンドウに何を載せ、何を載せないかを意図的に設計する技術 ── を重視する理由です（[Anthropic Engineering Blog](https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents)）。以降で紹介するスキルの設計指針は、すべてこのコンテキストエンジニアリングの考え方に基づいています。

## Claude Code スキルによるアプローチ

Claude Code のカスタムスキル機能を使えば、`/postmortem` のようなスラッシュコマンドを定義し、ポストモーテム準備の大部分を自動化できます。

### カスタムスキルとは

Claude Code のカスタムスキルは、`.claude/skills/<skill-name>/SKILL.md` に Markdown で定義するプロンプトテンプレートです。`/skill-name` で呼び出すと、そのプロンプトに基づいて Claude が動作します。

主な特徴は以下の通りです。

- **引数の受け渡し**: `$ARGUMENTS` で呼び出し時の引数を受け取れる
- **シェルコマンドの実行結果を埋め込み**: `` !`command` `` で事前にコマンドを実行し、結果をプロンプトに注入できる
- **サブエージェントの活用**: Agent ツールを使って並列に情報収集できる
- **MCP サーバーとの連携**: 設定済みの MCP サーバーを通じて外部サービスにアクセスできる

### 全体のフロー

`/postmortem` スキルの実行フローは次のようになります。

```
ユーザー: /postmortem INC-1234

1. スキルがインシデントIDを受け取る
2. 情報収集フェーズ（サブエージェントで並列実行）
   ├─ Agent A: Slack からインシデントチャンネルのログを取得
   ├─ Agent B: Datadog からアラート・メトリクスを取得
   ├─ Agent C: Jira からチケット情報を取得
   └─ Agent D: GitHub から関連 PR・Issue を取得
3. 要約・構造化フェーズ（別のサブエージェント）
   └─ 収集した情報を統合し、テンプレートに落とし込む
4. ポストモーテムのドラフトを出力
5. 人間によるレビューと加筆
```

ポイントは、**情報収集をサブエージェントで並列化**していることです。各データソースからの取得を同時に行うことで、逐次的にアクセスするよりも大幅に時間を短縮できます。

## スキルの設計指針

### 指針1: 情報収集はサブエージェントで並列化する

各データソースへのアクセスは独立しているため、サブエージェントで並列に実行するのが効果的です。

```markdown
## 情報収集

以下のサブエージェントを**並列**で起動し、情報を収集してください。

### Slack ログの収集
Agent ツールを使い、MCP 経由で Slack のインシデントチャンネル
(#incident-$ARGUMENTS) から以下を取得してください:
- インシデント発生から解決までの全メッセージ
- 対応者のアクションと判断のポイント

### Datadog アラートの収集
Agent ツールを使い、CLI (`datadog-ci`) 経由で Datadog から以下を取得してください:
- インシデント期間中に発火したアラートの一覧
- 関連するメトリクスの変化

### Jira チケットの収集
Agent ツールを使い、MCP 経由で関連する Jira チケットを取得してください。

### GitHub 変更の収集
Agent ツールを使い、CLI (`gh` コマンド) 経由で
インシデント前後の PR・デプロイを取得してください。
```

サブエージェントで並列化するメリットは、時間短縮だけではありません。前述した**コンテキスト爆発の問題を構造的に防げる**ことが最大のポイントです。

各データソースから取得するテキスト量は膨大になりがちです。Slack のインシデントチャンネルだけで数百メッセージ、Datadog のアラートログも数十件に及ぶことは珍しくありません。これらすべてをメインのコンテキストに流し込めば、たちまちコンテキストウィンドウが逼迫し、情報の取捨選択が雑になります。

サブエージェントは独立したコンテキストウィンドウを持ちます。つまり、各サブエージェントが膨大な生データを読み込み、**その中で要約・取捨選択を行った結果だけ**をメインのコンテキストに返します。これにより、メインのコンテキストは常にクリーンな状態を保てます。

#### CLIを優先し、MCPは必要な場合に使う

外部データソースへのアクセス手段として MCP サーバーと CLI の2つがありますが、2026年時点のベンチマークでは CLI の方がタスク完了率が28%高く、トークン効率も33%優れているという結果が出ています（[jannikreinhard.com](https://jannikreinhard.com/2026/02/22/why-cli-tools-are-beating-mcp-for-ai-agents/)）。

MCP サーバーはツール定義をすべてコンテキストに読み込むため、例えば GitHub の MCP サーバーは93ツールで約55,000トークンを消費します。一方、`gh` CLI は必要なコマンドだけを実行し、予測可能な出力を返します。

**推奨**: `gh`、`jira-cli`、`datadog-ci` などの CLI ツールが利用可能な場合はそちらを優先し、CLI では対応しにくいもの（Slack のメッセージ取得、Confluence のページ取得など）に MCP を使うのが実用的です。

#### サブエージェントの数は2〜4に抑える

Azure SRE Agent チームは、当初100以上のツールと50以上のサブエージェントで構築しましたが、**発見失敗**（能力はあるがルーティングが見つけられない）、**プロンプト脆弱性**（1つのエージェントの不調が全体に波及）、**無限ループ**（エージェント間でタスクをたらい回し）といった問題に直面しました。最終的に、少数のジェネラリストエージェントに集約することで劇的に改善したと報告しています（[Microsoft Tech Community](https://techcommunity.microsoft.com/blog/appsonazureblog/context-engineering-lessons-from-building-azure-sre-agent/4481200/)）。

Claude Code のサブエージェントも2〜4程度が適切です。それ以上に増やしても、オーケストレーションの複雑さが増すだけで改善は限定的です。

### 指針2: 収集した情報は別のサブエージェントで要約する

各サブエージェントが返す収集結果は、そのままでもある程度整理されていますが、データソースをまたいだ統合が必要です。収集結果を受け取った後、**別のサブエージェントに要約・構造化を担当させる**ことで、最終的な出力の品質を高められます。

```markdown
## 要約・構造化

収集した情報を以下の構造に整理してください:

### タイムライン
- 各データソースのタイムスタンプを突き合わせ、
  時系列順に「いつ、何が起きたか」を整理する
- 形式: `HH:MM` - イベント内容（情報源）

### 影響範囲
- 影響を受けたサービス・機能
- 影響を受けたユーザー数の推定
- ビジネスインパクト

### 対応アクション
- 誰が、いつ、何をしたか
- 判断のポイントとその根拠
```

ここでのポイントは、**収集と要約を別のサブエージェントに分ける**ことです。収集サブエージェントは「できるだけ多くの関連情報を拾う」ことに集中し、要約サブエージェントは「拾った情報を構造化して整理する」ことに集中します。責務を分離することで、それぞれのタスクの精度が上がります。

### 指針3: 出力を検証可能にする ── セルフレビューの限界を知る

AI にドラフトを生成させた後、「レビューして修正して」と指示すれば品質が上がると期待するかもしれません。しかし、RefineBench の評価（2025-2026）によると、LLM のセルフレビューによる改善は**5回の反復で+1.8ポイント以下**にとどまります。「何度もレビューさせれば品質が上がり続ける」というのは幻想です。

さらに、Zalando のポストモーテム分析（2年間・数千件の分析）では、高性能モデルでも約10%の「**表面帰属エラー**」（Surface Attribution Error）が残ることが報告されています ── インシデントに関連するキーワードが文中に出現するだけで、実際には因果関係がないのに因果関係があるかのような記述を生成してしまう現象です（[Zalando Engineering Blog, Sept 2025](https://engineering.zalando.com/posts/2025/09/dead-ends-or-data-goldmines-ai-powered-postmortem-analysis.html)）。

つまり、AI のセルフレビューには期待しすぎず、**人間が検証しやすい形で出力する**ことが重要です。

```markdown
## Step 4: 検証可能な形で出力する

タイムラインの各項目には、必ず情報源を明記してください:

| 時刻 | イベント | 情報源 |
|------|---------|--------|
| 10:23 | アラート発火 | Datadog: monitor-12345 |
| 10:25 | #incident-INC-1234 作成 | Slack: msg-ts-xxx |
| 10:32 | ロールバック PR マージ | GitHub: PR #456 |

人間がドラフトをレビューする際に、各記述の根拠を即座に
確認できることが最も重要です。
```

Zalando のチームは、当初100%の人間キュレーションを行い、徐々に10〜20%のサンプリングに移行しました。最初から完全に AI に任せるのではなく、信頼を段階的に構築するアプローチです。

### 指針4: テンプレートはチームの既存フォーマットに合わせる

ポストモーテムのテンプレートは組織ごとに異なります。スキルには自チームのテンプレートを埋め込んでおきましょう。

```markdown
## 出力フォーマット

以下のテンプレートに沿ってポストモーテムのドラフトを作成してください。

### タイトル
[日付] インシデント概要

### サマリー
- 何が起きたか（1-2文）
- 影響範囲

### タイムライン
| 時刻 | イベント | 情報源 |
|------|---------|--------|
| ...  | ...     | ...    |

### 影響
- 影響を受けたサービス
- 影響を受けたユーザー
- ダウンタイム

### 寄与要因（Contributing Factors）
（人間が記入）

### 再発防止策
（人間が記入）

### 教訓
（人間が記入）
```

一点、テンプレートの「根本原因」というセクション名については再考の余地があります。複雑な分散システムのインシデントでは、単一の根本原因が存在することはまれです。PagerDuty/Jeli の [Howie Guide](https://howie-guide.pagerduty.com/) や Learning from Incidents コミュニティでは、「Root Cause」ではなく「**Contributing Factors**（寄与要因）」という表現が推奨されています。何が起きたかを多角的に捉えるためには、「たった1つの原因」を探すのではなく、複数の寄与要因を列挙する方が実態に即しています。

### 指針5: 人間が担うべき部分を明示的に残す

ここが最も重要な設計指針です。

AI が得意なのは**情報の収集と整理**です。散在する情報源から関連データを引っ張ってきて、時系列に並べ、構造化する ── これは AI が圧倒的に速くこなせる作業です。

一方で、以下は人間が担うべきです。

- **寄与要因の分析**: なぜこのインシデントが起きたのか。技術的な要因だけでなく、組織やプロセスの要因も含めて多角的に分析する
- **再発防止策の策定**: 何をすれば再発を防げるか、優先度をつけて計画する意思決定
- **教訓の抽出**: チームとして何を学んだか、文化や仕組みにどう反映するか
- **ナラティブ（物語）の記述**: 対応者がどのような状況で判断を迫られ、何を考えていたか。テンプレートが捉えきれない「対応の物語」

スキルのテンプレートにこれらの項目を「（人間が記入）」と明示的に残しておくことで、AI と人間の役割分担が自然に成立します。incident.io のアプローチがまさにこれで、AI が構造化された80%を生成し、人間が10〜15分でナラティブと洞察を追加する形をとっています。

ポストモーテムの本質的な価値は、寄与要因の分析と再発防止策の議論にあります。準備の自動化は、チームがこの本質的な議論に集中するための手段です。

### 指針6: セキュリティを考慮する

AI エージェントに本番環境のモニタリングツールやコミュニケーションツールへのアクセスを与える際は、セキュリティに注意が必要です。

2025年の調査では、MCP 実装の43%にコマンドインジェクション脆弱性が発見されました（[Equixly Security Assessment](https://www.practical-devsecops.com/mcp-security-vulnerabilities/)）。また、MCP サーバーのツール説明文にユーザーから見えない悪意ある指示を仕込む「ツールポイズニング」攻撃も報告されています。

最低限、以下を守りましょう。

- **最小権限の原則**: エージェントに与える権限は読み取りのみに限定する。New Relic の SRE Agent は、意図的に「本番システムを変更しない、承認フローをバイパスしない、人間の判断を上書きしない」という制約を設けています
- **MCP サーバーの検証**: 信頼できるソースの MCP サーバーのみを使用し、ツール定義が変更されていないか定期的に確認する
- **自動承認を無効にする**: 本番ツールへのアクセスには、必ずユーザーの明示的な許可を求める設定にする

## スキルの実装エッセンス

以下は、`/postmortem` スキルの SKILL.md の骨格です。実際の環境に合わせて、MCP サーバーの設定やテンプレートをカスタマイズしてください。

```yaml
---
name: postmortem
description: インシデントIDを受け取り、各種データソースから情報を収集してポストモーテムのドラフトを生成する
user-invocable: true
disable-model-invocation: true
allowed-tools: Agent, Read, Write, Bash, Grep, Glob
argument-hint: [incident-id]
---
```

```markdown
あなたはポストモーテムの準備を支援するアシスタントです。
インシデントID: $ARGUMENTS

## Step 1: 情報収集（並列実行）

以下のサブエージェントを **並列** で起動してください。
各エージェントは、取得した情報を要約して返してください。
生データをそのまま返さないでください。

1. **Slack ログ収集**: #incident-$ARGUMENTS チャンネルの
   メッセージを取得し、主要なアクションと判断を時系列で要約
2. **アラート収集**: `datadog-ci` CLI でインシデント期間中の
   アラートを取得し、発火時刻と内容を一覧化
3. **チケット収集**: MCP 経由で関連する Jira チケットの
   ステータスと概要を取得
4. **コード変更収集**: `gh` CLI でインシデント前後の
   PR とデプロイを取得し、変更内容を要約

## Step 2: 要約・構造化

収集結果をもとに、タイムラインを時系列順に構成し、
影響範囲と対応アクションを整理してください。
```

## 注意: AI はトイルを増やすこともある

最後に、1つの注意点を述べます。

2025年の調査（[Runframe: State of Incident Management 2025](https://runframe.io/blog/state-of-incident-management-2025)）によると、AI 投資にもかかわらず運用トイルは25%から30%に上昇しました。AI ツールは、うまく統合されなければ、既存の作業を減らすどころか**検証のオーバーヘッド**を新たに生むリスクがあります。

ポストモーテムの準備スキルを導入した場合も、「AI の出力を人間が検証する」という新しい作業が発生します。この作業が、手作業で準備するよりも本当に軽いかどうかは、チームの環境と MCP サーバー／CLI の整備状況に大きく依存します。

まずは小さく始めて、実際に準備時間が短縮されるかを確認しながら、段階的にスキルを改善していくアプローチをおすすめします。Google Cloud の SRE チームが実践しているように、生成されたポストモーテムを次回以降の参考データとして蓄積する「フィードバックループ」を構築できれば、スキルの品質は時間とともに向上していきます。

## まとめ

ポストモーテムの準備は、情報源の多さとテキスト量の膨大さ、そしてインシデント直後の疲弊が重なって、非常に負担の大きい作業です。

コーディングエージェントの本質的な価値は、分断されたデータソースの情報を収集・統合できることにあります。ただし、スキルを適切に設計しなければ、一般論を表層的になぞるだけで終わったり、コンテキスト爆発で精度が落ちたりします。

この記事で紹介した6つの設計指針は、これらの問題に対処するためのものです。

1. **情報収集はサブエージェントで並列化する** ── コンテキスト爆発を構造的に防ぐ。CLI を優先し、サブエージェントは2〜4に抑える
2. **収集した情報は別のサブエージェントで要約する** ── 収集と整理の責務を分離する
3. **出力を検証可能にする** ── セルフレビューの限界を認識し、情報源の明記によって人間の検証を容易にする
4. **テンプレートはチームの既存フォーマットに合わせる** ── 「根本原因」より「寄与要因」を検討する
5. **人間が担うべき部分を明示的に残す** ── AI は情報収集、人間は分析・判断・ナラティブ
6. **セキュリティを考慮する** ── 最小権限、MCP サーバーの検証、自動承認の無効化

ポストモーテムの本質は、寄与要因を深掘りし、再発防止策を議論し、チームの学びとすることです。準備の自動化によって、チームがこの本質的な議論により多くの時間とエネルギーを使えるようになることを目指しています。
