## はじめに

インシデント対応の後にやってくるポストモーテム。対応そのものも大変ですが、ポストモーテムの**準備**もまた大きな負担です。

Slack のインシデントチャンネル、Datadog のアラートやダッシュボード、Jira のチケット、GitHub の Issue や Pull Request、Confluence のランブック ── 情報はあちこちに散らばっていて、それを時系列に整理し、テンプレートに落とし込む作業は、インシデント直後の疲弊した状態ではとにかく辛いものです。

この記事では、[Claude Code](https://docs.anthropic.com/en/docs/claude-code) のカスタムスキル機能を使って、ポストモーテムの準備作業を大幅に軽くするための設計指針を紹介します。

## ポストモーテム準備の何が辛いのか

ポストモーテム準備の辛さは、大きく4つに整理できます。

### 1. 情報源が多すぎる

インシデントに関する情報は、単一のツールに閉じていません。

- **コミュニケーション**: Slack のインシデントチャンネル
- **モニタリング**: Datadog のアラート・ログ・APM
- **タスク管理**: Jira のチケット
- **コード**: GitHub の Issue・Pull Request
- **ドキュメント**: Confluence のランブック・過去のポストモーテム

これらを横断的に見ながら「何が起きたか」を再構成する必要があります。

### 2. テキスト量が膨大

Slack のインシデントチャンネルには大量のメッセージが流れます。Datadog のアラートも複数発火しているでしょう。これらすべてに目を通し、関連する情報を拾い出すのは骨が折れます。

### 3. タイムライン作成が手間

「いつ、何が起きたか」の時系列を正確に組み立てるには、複数の情報源のタイムスタンプを突き合わせる必要があります。Slack のメッセージ、アラートの発火時刻、デプロイのタイムスタンプ ── これらを手作業で時系列に並べ替えるのは、単純だけれど時間のかかる作業です。

### 4. インシデント直後は疲弊している

そもそも、この準備作業はインシデント対応が終わった直後にやることが多いです。長時間の対応で疲弊した状態で、冷静に情報を整理するのは精神的にも体力的にもしんどいものです。

## Claude Code スキルによるアプローチ

Claude Code のカスタムスキル機能を使えば、`/postmortem` のようなスラッシュコマンドを定義し、ポストモーテム準備の大部分を自動化できます。

### カスタムスキルとは

Claude Code のカスタムスキルは、`.claude/skills/<skill-name>/SKILL.md` に Markdown で定義するプロンプトテンプレートです。`/skill-name` で呼び出すと、そのプロンプトに基づいて Claude が動作します。

主な特徴は以下の通りです。

- **引数の受け渡し**: `$ARGUMENTS` で呼び出し時の引数を受け取れる
- **シェルコマンドの実行結果を埋め込み**: `` !`command` `` で事前にコマンドを実行し、結果をプロンプトに注入できる
- **サブエージェントの活用**: Agent ツールを使って並列に情報収集できる
- **MCP サーバーとの連携**: 設定済みの MCP サーバーを通じて外部サービスにアクセスできる

### 全体のフロー

`/postmortem` スキルの実行フローは次のようになります。

```
ユーザー: /postmortem INC-1234

1. スキルがインシデントIDを受け取る
2. 情報収集フェーズ（サブエージェントで並列実行）
   ├─ Agent A: Slack からインシデントチャンネルのログを取得
   ├─ Agent B: Datadog からアラート・メトリクスを取得
   ├─ Agent C: Jira からチケット情報を取得
   └─ Agent D: GitHub から関連 PR・Issue を取得
3. 要約・構造化フェーズ（別のサブエージェント）
   └─ 収集した情報を統合し、テンプレートに落とし込む
4. ポストモーテムのドラフトを出力
```

ポイントは、**情報収集をサブエージェントで並列化**していることです。各データソースからの取得を同時に行うことで、逐次的にアクセスするよりも大幅に時間を短縮できます。

## スキルの設計指針

### 指針1: 情報収集はサブエージェントで並列化する

各データソースへのアクセスは独立しているため、サブエージェントで並列に実行するのが効果的です。

```markdown
## 情報収集

以下のサブエージェントを**並列**で起動し、情報を収集してください。

### Slack ログの収集
Agent ツールを使い、MCP 経由で Slack のインシデントチャンネル
(#incident-$ARGUMENTS) から以下を取得してください:
- インシデント発生から解決までの全メッセージ
- 対応者のアクションと判断のポイント

### Datadog アラートの収集
Agent ツールを使い、MCP または CLI 経由で Datadog から以下を取得してください:
- インシデント期間中に発火したアラートの一覧
- 関連するメトリクスの変化

### Jira チケットの収集
Agent ツールを使い、MCP 経由で関連する Jira チケットを取得してください。

### GitHub 変更の収集
Agent ツールを使い、CLI (`gh` コマンド) 経由で
インシデント前後の PR・デプロイを取得してください。
```

サブエージェントで並列化するメリットは、時間短縮だけではありません。各データソースから取得するテキスト量は膨大になりがちですが、サブエージェントは独立したコンテキストウィンドウを持つため、メインのコンテキストを圧迫しません。

### 指針2: 収集した情報は別のサブエージェントで要約する

各サブエージェントが返す生データはそのままでは量が多すぎます。収集結果を受け取った後、**別のサブエージェントに要約・構造化を担当させる**ことで、最終的な出力の品質を高められます。

```markdown
## 要約・構造化

収集した情報を以下の構造に整理してください:

### タイムライン
- 各データソースのタイムスタンプを突き合わせ、
  時系列順に「いつ、何が起きたか」を整理する
- 形式: `HH:MM` - イベント内容（情報源）

### 影響範囲
- 影響を受けたサービス・機能
- 影響を受けたユーザー数の推定
- ビジネスインパクト

### 対応アクション
- 誰が、いつ、何をしたか
- 判断のポイントとその根拠
```

### 指針3: テンプレートはチームの既存フォーマットに合わせる

ポストモーテムのテンプレートは組織ごとに異なります。スキルには自チームのテンプレートを埋め込んでおきましょう。

```markdown
## 出力フォーマット

以下のテンプレートに沿ってポストモーテムのドラフトを作成してください。

### タイトル
[日付] インシデント概要

### サマリー
- 何が起きたか（1-2文）
- 影響範囲
- 根本原因（判明している場合）

### タイムライン
| 時刻 | イベント | 情報源 |
|------|---------|--------|
| ...  | ...     | ...    |

### 影響
- 影響を受けたサービス
- 影響を受けたユーザー
- ダウンタイム

### 根本原因
（人間が記入）

### 再発防止策
（人間が記入）

### 教訓
（人間が記入）
```

### 指針4: 人間が担うべき部分を明示的に残す

ここが最も重要な設計指針です。

AI が得意なのは**情報の収集と整理**です。散在する情報源から関連データを引っ張ってきて、時系列に並べ、構造化する ── これは AI が圧倒的に速く、正確にこなせる作業です。

一方で、以下は人間が担うべきです。

- **根本原因の分析**: なぜこのインシデントが起きたのか、技術的・組織的な原因を深掘りする判断
- **再発防止策の策定**: 何をすれば再発を防げるか、優先度をつけて計画する意思決定
- **教訓の抽出**: チームとして何を学んだか、文化や仕組みにどう反映するか

スキルのテンプレートにこれらの項目を「（人間が記入）」と明示的に残しておくことで、AI と人間の役割分担が自然に成立します。ポストモーテムの本質的な価値は、根本原因の分析と再発防止策の議論にあります。準備の自動化は、チームがこの本質的な議論に集中するための手段です。

## スキルの実装エッセンス

以下は、`/postmortem` スキルの SKILL.md の骨格です。実際の環境に合わせて、MCP サーバーの設定やテンプレートをカスタマイズしてください。

```yaml
---
name: postmortem
description: インシデントIDを受け取り、各種データソースから情報を収集してポストモーテムのドラフトを生成する
user-invocable: true
allowed-tools: Agent, Read, Write, Bash, Grep, Glob
argument-hint: [incident-id]
---
```

```markdown
あなたはポストモーテムの準備を支援するアシスタントです。
インシデントID: $ARGUMENTS

## Step 1: 情報収集（並列実行）

以下の4つのサブエージェントを **並列** で起動してください。

1. **Slack ログ収集**: #incident-$ARGUMENTS チャンネルの全メッセージを取得
2. **アラート収集**: インシデント期間中の Datadog アラートを取得
3. **チケット収集**: 関連する Jira チケットの情報を取得
4. **コード変更収集**: `gh` CLI でインシデント前後の PR を取得

## Step 2: 要約・構造化

収集結果をもとに、タイムラインを時系列順に構成し、
影響範囲と対応アクションを整理してください。

## Step 3: ドラフト出力

チームのテンプレートに沿ってポストモーテムのドラフトを出力してください。
「根本原因」「再発防止策」「教訓」は人間が記入する欄として空けてください。
```

## まとめ

ポストモーテムの準備は、情報源の多さとテキスト量の膨大さ、そしてインシデント直後の疲弊が重なって、非常に負担の大きい作業です。

Claude Code のカスタムスキルを使えば、以下のように準備の負担を軽減できます。

- **サブエージェントの並列実行**で、複数のデータソースから同時に情報を収集する
- **別のサブエージェントで要約・構造化**し、膨大なテキストを扱いやすい形にまとめる
- **チームのテンプレート**に沿ったドラフトを自動生成する
- **人間が担うべき部分**を明示的に残し、AI と人間の役割分担を明確にする

ポストモーテムの本質は、根本原因を深掘りし、再発防止策を議論し、チームの学びとすることです。準備の自動化によって、チームがこの本質的な議論により多くの時間とエネルギーを使えるようになることを目指しています。
