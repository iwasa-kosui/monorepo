今月の通底テーマは「AI加速の副作用を運用工学で抑える」流れだった。OpenTelemetryがCNCFを卒業し観測領域の事実上の標準として地位を確定させた一方、Martin Fowler は VibeCoding と VibeSec の連作でAI駆動開発のセキュリティ・保守性リスクを正面から扱い、Kubernetes v1.36 "Haru" は User Namespaces と DRA を中心にAIワークロード本番化の足回りを整えた。クラウドベンダー側では AWS の AgentCore 続報、Google I/O '26 の ADK 2.0、Microsoft Build 2026 直前の WSL 3 / Azure Agent Mesh と、エージェント実行基盤の主導権争いが連続して可視化された。

## 設計パターン・アーキテクチャ思想

### Martin Fowler: VibeCoding と VibeSec の連作

[VibeCoding](https://martinfowler.com/bliki/VibeCoding.html)（5月21日）/ [The VibeSec Reckoning](https://martinfowler.com/articles/vibesec-reckoning.html)（5月27日）/ [Maintainability sensors for coding agents](https://martinfowler.com/articles/sensors-for-coding-agents.html)（5月19日）/ [What is Code](https://martinfowler.com/articles/what-is-code.html)（5月12日）

- VibeCoding は LLM への指示だけでコードを確認せずアプリを作る手法を整理し、プロトタイピング加速の利点と保守性・正確性・セキュリティの劣化を対比した
- VibeSec はセキュリティリスクの実体験レポートで、セキュリティコンテキストファイルの整備と日次セキュリティインテリジェンスフィードの組み合わせを推奨
- Maintainability sensors はコーディングエージェント出力を静的解析・リンターで継続計測しフィードバックループに組み込む手法
- What is Code はコードを「機械への指示」と「問題領域の概念モデル」の2面から再定義し、LLM時代の言語設計を論じている

### Martin Fowler: SPDD（Structured-Prompt-Driven Development）

[Structured-Prompt-Driven Development](https://martinfowler.com/articles/structured-prompt-driven/)（4月28日）

LLMプロンプトをバージョン管理対象の第一級成果物として扱う方法論。要件から実装を体系化する REASONS Canvas（Requirements / Entities / Approach / Structure / Operations / Norms / Safeguards の7要素）を導入し、Abstraction First・Alignment・Iterative Review の3スキルを軸にしている。Tech Radar Vol.34 の Caution「セマンティック拡散」への対抗策にあたる。

### InfoQ: 主権障害ドメインとAdaptive Hedged Requests

[Sovereign Fault Domains](https://www.infoq.com/articles/sovereign-fault-domains-cloud-resilience/)（4月22日）/ [Adaptive Hedged Requests](https://www.infoq.com/articles/adaptive-hedged-requests-p99-latency/)（5月28日）

- Sovereign Fault Domains は地政学的境界（法的管轄・経済制裁・インターネット遮断）を障害境界として設計に組み込み、マルチAZではなくマルチリージョンを既定構成として提案する
- Adaptive Hedged Requests は失敗リトライではなく遅延中のプライマリと並行してバックアップを走らせ先着優先で p99 を74%削減する手法。DDSketch によるリアルタイム分位数追跡とトークンバケットで負荷増加を抑制する

### InfoQ: Schema Proliferation と Cloud-Native Kafka

[Schema Proliferation 問題](https://www.infoq.com/articles/schema-proliferation-problem/)（5月25日）/ [Architecting Cloud-Native Kafka](https://www.infoq.com/articles/architecting-cloud-native-kafka/)（5月26日）

- Kafka-Flink パイプラインで増殖するスキーマを判別子ベースの統合スキーマで吸収する設計を提示。12スキーマを1つに統合し保守コストと後方互換性を両立
- 階層化ストレージからディスクレス構成へというクラウドネイティブKafkaの設計変化を整理

### Event Sourcing / DDD コミュニティ

[EventStoreDB / Axon / Polecat 比較記事](https://dasroot.net/posts/2026/04/event-sourcing-cqrs-databases-eventstoredb-axon-polecat/)（4月10日）/ [DDD 実践シリーズ Part 3](https://www.endpointdev.com/blog/2026/05/designing-software-architecture-ddd-part-3/)（5月）

- Axon 5.1.0-RC2 でバッチ処理最適化によりリプレイ速度が260→4,000イベント/秒に。EventStoreDB は Kurrent に改称
- DDD 実践シリーズ Part 3 はレイヤード・ポーツ＆アダプター・CQRS・Outbox・Saga/Process Manager を一体的に扱う

## まとめ

エージェント実行基盤（認証・決済・サンドボックス・ガバナンス）の整備が AWS（AgentCore）・Google Cloud（ADK 2.0 + Agent Registry）・Microsoft（Azure Agent Mesh）の3社共通テーマとなり、ランタイム層・オーケストレーション層・エッジ分散実行層と棲み分けが見え始めた。Uber のエージェントアイデンティティ管理や Spotify の広告マルチエージェントのように、エージェントを本番インフラの一部として扱う事例も具体化している。一方で Martin Fowler の VibeSec や ThoughtWorks の認知的負債が示すとおり、AI加速の副作用を観測可能性（OpenTelemetry GenAI 規約・Honeycomb Agent Observability）と運用工学で抑える方向の議論が同時進行している。
