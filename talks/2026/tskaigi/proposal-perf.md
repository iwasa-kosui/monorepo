## 概要（Abstract）

「Node.jsはシングルスレッドだから遅い」——そんな漠然とした理解のまま、パフォーマンス問題に直面したとき、あなたはどこから調査を始めますか？

カケハシでは、医療SaaSの認証基盤をフルスタックTypeScript（Node.js）で運用しています。本セッションでは、本番環境で実際に発生した2つのパフォーマンス問題について、Datadog APMのContinuous Profilerを用いた原因特定から解決までのプロセスを、Node.jsランタイムの内部動作と**TypeScriptの型システムを活かした解決パターン**を交えながら解説します。

**事例1: CPU Bounded処理によるイベントループのブロッキング**
リクエストのレイテンシが突然急増。CPUプロファイルを取得すると、パスワードハッシュ化（argon2id）やSSRのDOMレンダリングがイベントループを長時間占有していたことが判明しました。シングルスレッドモデルにおけるCPU Bounded処理の影響と、Worker Threadsや非同期化による解決策に加え、`AsyncLocalStorage<T>`とTypeScriptジェネリクスによる型安全なリクエストコンテキスト管理を紹介します。

**事例2: ヒープ溢れとGC頻発によるCPU占有**
CPU使用率が恒常的に高い状態が続く。ヒーププロファイルを確認すると、外部サービスクライアントがリクエストごとに生成され、ヒープが溢れていました。V8のGC（Scavenge, Mark-Sweep-Compact）が頻発し、アプリケーション本来の処理にCPUが割けない状態に。クライアントのシングルトン化とメモリ設定の調整に加え、TypeScriptの型推論を活用した軽量シングルトンパターンで解決に至ったプロセスを紹介します。

これらの事例を通じて、「パフォーマンス問題の発見→Node.js内部の理解→TypeScriptらしい解決」という一貫したプロセスと、問題発生時に「何を見ればよいか」を持ち帰っていただけるセッションを目指します。

## トーク構成（30分）

### 第1部: 背景（4分）

#### カケハシの認証基盤とフルスタックTypeScript（2分）

- 医療SaaS向けの認証基盤をNode.js（TypeScript）で構築・運用
- なぜNode.jsを選んだのか：フロントエンドとの技術スタック統一、エコシステムの充実
- 一方で、Node.js特有の注意点がある

#### Node.jsのシングルスレッド・イベントループモデル（2分）

- イベントループの基本構造：コールスタック、タスクキュー、マイクロタスクキュー
- 非同期I/Oは得意だが、CPU Bounded処理はイベントループをブロックする
- V8のメモリ管理の詳細は事例2で解説

### 第2部: 事例1 — CPU Bounded処理によるブロッキング（10分）

#### 症状の発見（2分）

- リクエストのp99レイテンシが急増
- 特定のエンドポイントだけでなく、サーバ全体に影響が波及
- Datadogのダッシュボードで異常を検知

#### 原因調査：Continuous Profiler（3分）

- Datadog APMの`@datadog/pprof`によるCPUプロファイル取得
- フレームグラフの読み方と、ボトルネックの特定プロセス
- 原因: パスワードハッシュ化（argon2id）やSSRのDOMレンダリングがイベントループを長時間占有

#### Node.js内部の解説（2分）

- CPU Bounded処理がイベントループに与える影響の可視化
- なぜ1つの重い処理が他の全リクエストに影響するのか
- libuvのスレッドプールとの関係

#### 解決策とTypeScriptパターン（3分）

- Worker Threadsへのオフロード
- 非同期ハッシュ化ライブラリの利用
- SSRの見直し：CSRへの段階的移行
- **AsyncLocalStorage + TypeScriptジェネリクスで型安全なリクエストコンテキスト**
  - 調査時の課題：リクエストIDやユーザーIDをロギングに含めたいが、引数のバケツリレーはコードが肥大化しパフォーマンスの見通しも悪くなる
  - `AsyncLocalStorage<T>`でリクエストスコープのコンテキストを暗黙的に伝播
  - TypeScriptのジェネリクスでストアの型を厳密に定義し、型安全にアクセス
  - パフォーマンスへの影響が小さく、オブザーバビリティを損なわない設計
- 解決後のメトリクス変化

### 第3部: 事例2 — ヒープ溢れとGC頻発（12分）

#### 症状の発見（2分）

- CPU使用率が恒常的に高い状態
- リクエスト数に対してCPU消費が不釣り合い
- レスポンスタイムのばらつきが大きい（GCのStop-the-World）

#### 原因調査：ヒーププロファイル（3分）

- Datadog APMのヒーププロファイルでメモリ使用状況を可視化
- アロケーションの多いコードパスの特定
- 原因: 外部サービスクライアント（HTTPクライアント、SDKインスタンス等）がリクエストごとに生成

#### V8メモリ管理とGCの内部動作（3分）

- V8のメモリ管理：New Space / Old Space の構造
- V8のGCの仕組み：Scavenge（New Space）とMark-Sweep-Compact（Old Space）
- GC頻発がCPUを占有するメカニズム
- `--max-old-space-size`とGC閾値の関係

#### 解決策とTypeScriptパターン（4分）

- クライアントのシングルトン化（モジュールスコープでの初期化）
- `--max-old-space-size`の適切な設定
- **TypeScriptの型推論を活用した軽量シングルトン化関数**
  - classベースのGoFシングルトンパターンの問題点：インスタンス管理のボイラープレート、テスト時のリセット困難
  - モジュールスコープ + クロージャ + 型推論で実現する軽量パターン
  - `as const satisfies`や関数の戻り値型推論でファクトリの型を厳密に保つ
  - クラスインスタンスよりプレーンオブジェクトの方がV8にとって軽量であることの解説
- メモリリークの継続的モニタリング
- 解決後のメトリクス変化

### 第4部: 学びとまとめ（4分）

#### Node.jsサーバサイドのパフォーマンスの勘所（2分）

- イベントループをブロックしない設計の重要性
- オブジェクトのライフサイクルを意識する（リクエストスコープ vs アプリケーションスコープ）
- 「計測なき最適化は悪」：プロファイリングから始める
- **「TypeScriptの型システムはパフォーマンスにも貢献する」** — 型安全な設計が不要なアロケーションやバグを防ぐ

#### 調査ツール・手法の整理（2分）

- Datadog APMのContinuous Profiler（CPU / Heap）
- `--inspect`とChrome DevTools
- Node.jsの`--prof`フラグ
- 問題の種類に応じたツール選択の指針

## 想定される質疑応答

- **「Worker Threadsの導入はどの程度複雑か？」** → 認証基盤のケースでは、パスワードハッシュ化のような独立した処理をWorker Threadsに移すのは比較的容易でした。ただし、共有メモリの扱いやエラーハンドリングには注意が必要です。具体的な実装パターンをセッション内で紹介します
- **「Datadog以外のツールでも同様の調査は可能か？」** → はい。Node.jsの`--prof`フラグやChrome DevToolsの`--inspect`でも同様のプロファイリングが可能です。Datadogの利点は本番環境で常時プロファイルを収集できる点と、APMトレースとの紐付けが容易な点です
- **「シングルトン化によるデメリットはないか？」** → テストのモック差し替えが難しくなるケースがあります。DI（依存性注入）パターンと組み合わせることで、テスタビリティを損なわずにシングルトン化できます
- **「AsyncLocalStorageのパフォーマンスオーバーヘッドはどの程度か？」** → Node.js 16以降では内部実装が最適化されており、ベンチマーク上のオーバーヘッドは無視できるレベルです。引数のバケツリレーによるコード肥大化やミスのリスクと比較すると、総合的にメリットが上回ります
- **「classベースのシングルトンと関数ベースの軽量パターンはどう使い分けるか？」** → 状態の変更やライフサイクル管理が必要な場合はclassが適しています。SDKクライアントのように初期化後は不変のオブジェクトであれば、モジュールスコープ + クロージャの軽量パターンの方がV8にとっても効率的です
- **「SSRをやめてCSRに移行する判断基準は？」** → 認証基盤のケースでは、SEOが不要な管理画面系のページから段階的にCSRへ移行しました。ユーザー体験とパフォーマンスのトレードオフを定量的に評価することが重要です
