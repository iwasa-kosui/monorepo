## 概要（Abstract）

Node.jsのパフォーマンス問題の多くは、リソースのライフサイクル管理の誤りに起因します。リクエストごとに重いクライアントを生成してヒープを溢れさせる、CPU Bounded処理でイベントループを止める——これらは「どのオブジェクトが、いつ生まれ、いつ消えるか」が制御できていないことの帰結です。

本セッションでは、認証基盤の本番運用で遭遇した2つの事例をもとに、V8のGCやイベントループの内部動作を解説した上で、TypeScriptの型システムとES2025の`using`宣言でリソースライフサイクルをコンパイル時に強制するアプローチを提案します。

**事例1: CPU Bounded処理とイベントループ**
argon2idハッシュ化とSSRがイベントループを占有。Worker Threadsへのオフロードに加え、Node.js 24で標準となったAsyncContextFrameベースの`AsyncLocalStorage<T>`を用いた型安全なリクエストコンテキスト伝播を紹介します。

**事例2: リクエストスコープのヒープ爆発とGC**
外部クライアントの無秩序な生成がGCを頻発させCPUを圧迫。ES2025の`using`宣言（Explicit Resource Management）でリソースの破棄を型レベルで強制し、V8のHidden Classを意識したオブジェクト設計でGC負荷を構造的に低減するパターンを解説します。

## トーク構成（30分）

### 第1部: リソースライフサイクルという視点（3分）

- 認証基盤の紹介と、2つのパフォーマンス障害の概要
- 共通の根本原因：リソースのライフサイクル（生成・保持・破棄）が管理されていない
- 本セッションのテーマ：V8の動作を理解し、TypeScriptで予防する

### 第2部: 事例1 — CPU Bounded処理とイベントループ（10分）

#### 症状と調査（4分）

- p99レイテンシ急増 → Datadog Continuous Profilerで原因特定
- フレームグラフの読み方：argon2id・SSR DOMレンダリングがイベントループを占有
- イベントループのフェーズ（timers → poll → check）とCPU Bounded処理がブロックする仕組み

#### 解決と発展（6分）

- Worker Threadsオフロード / CSR移行で直接的に解消
- **AsyncLocalStorage<T>とAsyncContextFrame**
  - 調査時に欲しかったリクエストコンテキスト（リクエストID、ユーザーID）の伝播
  - Node.js 24でデフォルトとなったAsyncContextFrame実装：async_hooksベースからV8ネイティブへの移行とパフォーマンス改善
  - TC39 AsyncContext提案（Stage 1）との関係と将来の標準化
  - TypeScriptジェネリクスでストアの型を厳密に定義し、`getStore()`の戻り値を型安全に

### 第3部: 事例2 — ヒープ爆発とGC（13分）

#### 症状と調査（3分）

- CPU使用率の恒常的な高騰、レスポンスタイムのばらつき
- ヒーププロファイルで外部クライアントのリクエストスコープ生成を特定

#### V8のメモリ管理とGC（4分）

- New Space（Scavenge）/ Old Space（Mark-Sweep-Compact）の役割
- 短命オブジェクトの大量生成がScavengeを頻発させるメカニズム
- **Hidden Class（Map）とインラインキャッシュ**：V8はオブジェクトの「形状」を追跡して最適化する。TypeScriptのインターフェースに沿った一貫したオブジェクト形状が、JIT最適化（Maglev → TurboFan）を助ける

#### 解決：型でリソースライフサイクルを強制する（6分）

- クライアントのシングルトン化 + `--max-old-space-size`調整で直接的に解消
- **ES2025 `using`宣言（Explicit Resource Management）によるリソース管理**
  - `Symbol.dispose` / `Symbol.asyncDispose`でリソースの破棄を型レベルで強制
  - `using`を付け忘れるとTypeScriptがコンパイルエラーを出す — リソースリークをランタイムではなくコンパイル時に検出
  - DB接続、HTTPクライアント、ファイルハンドルなど、認証基盤での具体的な適用例
- **V8 Hidden Classを意識したオブジェクト設計**
  - classインスタンスとプレーンオブジェクトのV8内部表現の違い
  - `as const satisfies`でオブジェクト形状を型レベルで固定し、Hidden Classの安定性を高める

### 第4部: まとめ（4分）

- 2つの事例の共通構造：ランタイムの問題を型システムで予防する
- リソースライフサイクルの3層：アプリケーションスコープ / リクエストスコープ / `using`スコープ
- Node.js 24 + TypeScript 5.8 + ES2025が揃い、型による予防的パフォーマンス設計が現実的になった

## 想定される質疑応答

- **「AsyncContextFrameの導入でAsyncLocalStorageのパフォーマンスはどう変わったか？」** → async_hooksベースでは各非同期操作でフックが発火していたのに対し、AsyncContextFrameはデータ変更時のみ新フレームを生成するため、オーバーヘッドが大幅に減少しました。Node.js 24ではデフォルトで有効です
- **「`using`宣言は実際にどの程度パフォーマンスに寄与するか？」** → `using`自体は直接的な高速化ではなく、リソースリークの防止が目的です。リークが起きなければGC負荷が安定し、結果としてパフォーマンスが予測可能になります
- **「Hidden Classの安定性はベンチマークで確認できるか？」** → V8の`--trace-maps`フラグでMap遷移を可視化できます。セッション内でプレーンオブジェクト vs 動的プロパティ追加のMap遷移の違いをデモします
