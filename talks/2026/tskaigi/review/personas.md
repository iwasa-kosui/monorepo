# 聴講者ペルソナ

TSKaigi 2026 登壇「TypeScriptのclassはなぜこうなったのか」の聴講者像を整理する。プロポーザルが掲げる対象者像、TSKaigi の参加者構成、フレームワーク依存度・言語経験の多様性を踏まえ、批判レビューの「観点」を抽出するためのペルソナを定義する。

各ペルソナはスライドを最初から最後まで通しで聴いた前提で、どこで詰まり・どこで物足りなく感じ・どこで反発するかを引き出すための装置である。実在の特定個人を指すものではない。

---

## P1. 高木さん（32歳）— Java から NestJS に移った中堅バックエンド

### 経歴・現職

- 新卒で Java/Spring を 7 年。直近 3 年は Node.js+NestJS でサーバーサイド開発
- 業務系 SaaS のドメイン層を担当。TypeORM の Entity に `@Column` などを日常的に書く
- TypeScript の `class` を書く時間が長く、Java の感覚を引きずっている自覚はある

### 言語観

- 「型はクラス名で決まるもの」「`private` はアクセス制御」「`instanceof` は型判定」が体に染みついている
- 関数型は触れたことがあるが、業務では使っていない

### 期待

- 日常的に書く `class` の「なんか変だな」の正体を知りたい
- 自分のチームの NestJS コードベースを、明日からどう書き換えるべきか持ち帰りたい

### 抱える疑問・想定する反発

- 「Branded Type は良さそうだが、`@Entity()` を付けた TypeORM クラスにも本当に効くのか？」
- 「`#private` に全部置き換えるべきなのか？ TypeORM の reflect ベースの仕組みと衝突しないか？」
- 「コンパニオンオブジェクトは美しいが、NestJS の DI とどう同居させるのか」

### この登壇への評価軸

- 落とし穴の説明が「自分のコード」に直結するか
- 対策がフレームワーク現実と整合しているか
- 「class をやめろ」と言われていると受け取れる箇所がないか

## P2. 西村さん（29歳）— React/Next.js の関数型志向フロントエンド

### 経歴・現職

- 新卒からフロントエンド一筋 5 年。React の関数コンポーネント以降しか知らない
- 業務で `class` を書くのは Error Boundary くらい。それも年に数回
- fp-ts / Effect / Zod を業務に持ち込んでいるタイプ

### 言語観

- `class` は「歴史的経緯で残っているが避けるべきもの」と素朴に思っている
- Discriminated Union と Branded Type は当然のように使う

### 期待

- 「なぜ class がここまで生き残っているのか」を理解したい
- 「使わなくていい根拠」を補強できるなら持ち帰りたい
- 一方で、本当に必要な場面があるなら謙虚に学びたい

### 抱える疑問・想定する反発

- 「対策篇で示される代替案、自分はもう全部やっている。新規性はあるのか？」
- 「フレームワークが class を要求する例は Angular/NestJS だけ。30 分の半分をその話に割くのは過剰では？」
- 「Web Components が出てくるが、業務で書いているフロントエンドエンジニアはどれくらいいるのか」

### この登壇への評価軸

- 「class を使わない側」にも 30 分聴く価値が示されているか
- 歴史パートが「単なる雑学」ではなく現在の設計判断に効いているか

## P3. 佐藤さん（25歳）— NestJS で実装するが言語仕様は未習得のジュニア

### 経歴・現職

- 社会人 2 年目。最初の現場が NestJS+Prisma で、TypeScript はそこで覚えた
- 言語仕様より「フレームワークの作法」で書いている自覚あり
- `prototype` という単語を聞いてもピンと来ない

### 言語観

- `class` と `interface` の違いをコードジェネレータの違いくらいに捉えている
- `useDefineForClassFields` は名前すら聞いたことがない

### 期待

- 自分のチームの先輩が話している「落とし穴」をついていけるようになりたい
- 落とし穴を踏む前に避ける引き出しを増やしたい

### 抱える疑問・想定する反発

- 「`[[Define]]` vs `[[Set]]` の表記の意味が分からない。何を読めばいいのか」
- 「prototype チェーンのシンタックスシュガー、と言われても素のイメージが湧かない」
- 「Branded Type のコードはコピペできるが、なぜ `Symbol` を使うのか分からない」

### この登壇への評価軸

- 仕様用語の説明に「初出のジュニアでも掴める導線」があるか
- 落とし穴と対策が「コピペで明日から使える」レベルまで降りているか
- ついていけなくなる「断崖」がどこにあるか

## P4. 朝倉さん（38歳）— Rust/Haskell 経由でテックリードに着任したシニア

### 経歴・現職

- Rust と Haskell を 10 年。最近 1 年は会社の都合で TypeScript のフルスタック領域を担当
- 言語仕様や spec の議論を一次資料で追うのが好き。TC39 のミーティングノートも時々読む
- TypeScript の `class` を意図的に避けて関数型でドメインを書いている

### 言語観

- 「nominal vs structural」「erasure semantics」を母語のように扱う
- 歴史的経緯と設計判断には強い興味がある

### 期待

- 一次資料に裏付けられた歴史記述と、設計トレードオフの言語化を聴きたい
- 自分が直感的に避けている `class` の理由を、人に説明できる形で言語化したい

### 抱える疑問・想定する反発

- 「ES4 の挫折を Microsoft の反対だけで説明するのは粗い。Macromedia/Mozilla 側の主張は？」
- 「`maximally minimal classes` を Allen Wirfs-Brock の名で出すなら、そのモチベーション（後方互換性・実装コスト）まで踏み込んでほしい」
- 「`[[Define]]` vs `[[Set]]` を tsconfig フラグの話に閉じ込めず、`Object.defineProperty` のセマンティクスとして説明したほうが芯を食う」
- 「Branded Type を `Symbol` で実装している例があるが、`unique symbol` でない以上、型レベルで本当に分離できているのか怪しい」

### この登壇への評価軸

- 一次資料・仕様書レベルの正確性が担保されているか
- 設計判断の言語化が「結論ありき」ではないか
- スライドのコード例が仕様的に厳密か

## P5. 三浦さん（35歳）— 型システムが大好きな OSS ライブラリ作者

### 経歴・現職

- 日中はプロダクト開発、夜は型パズル系の OSS をメンテ
- `npm publish` する側の視点を持ち、ライブラリの API 表面に潔癖
- 自分のライブラリで `class` を export するかどうか、毎回真剣に悩む

### 言語観

- `branded`, `phantom type`, `nominal` といったキーワードを日常的に使う
- `#private` と `private` の違いは「ユーザーが触れる API 表面の話」として整理している

### 期待

- ライブラリ作者として「class を export する/しない」の判断軸を補強したい
- Branded Type の実装パターンを、自分の OSS で採用可能か検討したい

### 抱える疑問・想定する反発

- 「`Brand<K, T extends symbol> = K & { [k in T]: true }` のパターンは古い。Zod v4 や `unique symbol` ベースの近年のパターンに触れないのは片手落ちでは？」
- 「`Symbol()` を毎回新規生成すると、同じモジュールを別ファイルから import した時に型が分岐しないか？」
- 「`#private` フィールドは継承・WeakMap・Proxy で扱いが面倒。ライブラリ作者視点での落とし穴を一言ほしい」
- 「Stage 3 decorators の話は触れているが、ライブラリ作者が今日採用すべきかの判断材料がない」

### この登壇への評価軸

- 提示される実装パターンが 2026 年時点で最新か
- ライブラリ公開 API としての観点（後方互換・型推論の漏れ）があるか
- 「中級者向け」の名のもとに、本来踏み込むべき細部を端折っていないか

## ペルソナごとの観点マトリクス

各ペルソナがスライドのどのパートを最も厳しく見るかを整理する。これに沿って次の `critique.md` で批判を展開する。

- P1 高木（NestJS）: 対策篇のフレームワーク現実、Branded Type と TypeORM の整合
- P2 西村（React 関数型）: 歴史パートと対策篇の自分への新規性、フロントエンド事例の薄さ
- P3 佐藤（ジュニア）: 用語の初出説明、prototype・`[[Define]]` などのジャンプ、コード例の補助線
- P4 朝倉（Rust/Haskell）: 一次資料の正確性、設計トレードオフの言語化、仕様用語の厳密性
- P5 三浦（OSS）: Branded Type 実装の最新性、`#private` のライブラリ視点、Stage 3 decorators の採用判断
