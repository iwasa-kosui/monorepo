# ペルソナ別 批判レビュー（Round 1）

`personas.md` の 5 ペルソナ視点で、`slides.md`（41,247 bytes / 約 60 スライド相当）を通しでレビューする。各指摘には次を付す:

- **箇所**: スライドの該当行または論点
- **指摘**: 何が問題か（根拠つき）
- **優先度**: 🔴 致命 / 🟡 重要 / 🟢 改善余地
- **改善方針**: スライド改修の方向

「致命」は登壇内容の正確性・論理整合に関わるもの、「重要」は対象ペルソナの満足度を著しく損なうもの、「改善余地」はあれば良いもの、と区別する。

## P1. 高木さん（Java→NestJS）視点

### P1-1 🔴 「2 つの根本原因」に thisバインディングが収まっていない

- **箇所**: スライド 790-808 行「2 つの根本原因」、869 行以降「落とし穴 2: thisバインディング」
- **指摘**: スライドは「4 つの落とし穴はすべて構造的部分型と型消去に帰着する」と宣言しているが、落とし穴 2（this）の根本原因はスライド本体（900-921 行）で「prototype ベース OOP の動的束縛」と説明されている。これは構造的部分型でも型消去でもなく、3 つ目の独立原因。NestJS で `@Injectable()` クラスのメソッドを Express ミドルウェアに直接渡してハマるのは日常的にあり、看板と中身が食い違うと「では `this` 問題はどう体系化するのか」と置いてけぼりになる。
- **改善方針**: 「2 つの根本原因」を「3 つの根本原因（構造的部分型 / 型消去 / prototype の動的 this）」に拡張するか、もしくは this を「型システムではなく実行時セマンティクスの問題」と章立てを分離する。プロポーザル本文 31 行も「2 つの根本原因」と書いているため、proposal.md との整合も必要。

### P1-2 🟡 Branded Type の対策が「class を捨てる例」に偏り、NestJS で適用できる像が浮かばない

- **箇所**: 1077-1124 行「対策 1: Branded Types / Zod スキーマとの組み合わせ」
- **指摘**: コード例の `User` / `Product` は `Readonly<{ id: UserId; name: string }>` と型エイリアスで書かれており、`class` を捨てた前提になっている。NestJS の `@Entity()` クラスや `@Injectable()` サービスを書き続ける高木にとっては、「自分の Entity の `id: string` を Branded Type に置き換えていいのか」「TypeORM の reflect metadata と衝突しないか」がわからない。プロポーザルの差別化ポイント（クラスをやめろではない）にも反する。
- **改善方針**: Branded Type のスライドに「class の中で使うパターン」を 1 枚追加する。`class User { constructor(public id: UserId, public name: string) {} }` のように、class フィールドの型として Branded Type を採用する例。

### P1-3 🟡 「フレームワークと class の現実」が一覧表で終わっており、対処の翻訳がない

- **箇所**: 1046-1069 行「フレームワークと class の現実」
- **指摘**: 表は事実列挙にとどまり、「NestJS でこの後紹介する対策をどう使うか」の翻訳がない。NestJS は Stage 3 decorator にまだ未対応（experimentalDecorators 必須）で、`#private` も TypeORM/class-transformer と相性問題がある。「class を使い続ける以上、4 つの落とし穴のうち何を諦め、何で殴れるか」の整理が必要。
- **改善方針**: 「NestJS で各落とし穴にどう向き合うか」の小マトリクスを 1 枚足す。たとえば: 構造的部分型 → Branded Type で殴れる / this → アロー関数フィールド or `.bind()` / private → TS private で当面妥協 / instanceof → DTO は Zod で殴る。

### P1-4 🟢 アロー関数フィールドのトレードオフが「メモリ影響」で止まっており、NestJS 観点が薄い

- **箇所**: 1136-1169 行「対策 2: thisバインディング」
- **指摘**: アロー関数フィールドが prototype に載らない件は触れているが、NestJS の `@Injectable()` Singleton 文脈では「prototype に載らないと AOP 系 interceptor が掛けられない」のが具体的な痛みになる。汎用的な「メモリ影響」より具体例の方が中堅エンジニアには刺さる。
- **改善方針**: 余白があれば一言「decorator/interceptor で AOP したいメソッドはアロー関数フィールドにしてはいけない」を追記。

## P2. 西村さん（React/関数型）視点

- **改善方針**: まとめスライド（1248 行）に「class を使っていない人への持ち帰り」を 1 行追加する。たとえば「フレームワークが要求する class を読み解く・批評する語彙が得られる」「Branded Type / Discriminated Union を採用済みなら、なぜそれが選ばれたかの歴史的根拠を持てる」。

### P2-2 🟢 Web Components の扱いが浅く、フロントエンド開発者に響かない

- **箇所**: 1056 行のフレームワーク表「Web Components: `class extends HTMLElement` が仕様上必須」

- **改善方針**: 行数の都合で表自体は変えづらいが、ノートに一言補足する。

### P2-3 🟡 歴史パートと対策篇の橋渡しが弱い

- **箇所**: 762-776 行「歴史篇まとめ」、1040-1043 行「3. 対策篇」セクション
- **指摘**: 歴史篇まとめが「複雑さは歴史的産物」で終わり、対策篇は突如「フレームワークと class の現実」に入る。歴史を聴いた直後、「では歴史を踏まえて自分はどう振る舞うか」の橋がない。西村のような「歴史だけ聴きに来た訳ではない」聴衆にとって、章間の動機づけが弱い。
- **改善方針**: 落とし穴篇の冒頭（778 行 / 「4 つの落とし穴はなぜ生まれるのか」）の前か、対策篇の冒頭に「歴史を踏まえると、これから紹介する落とし穴が偶然ではなく必然だとわかる」という橋渡しを 1 枚足す。

## P3. 佐藤さん（ジュニア）視点

### P3-1 🔴 Q3 のクイズが「正解できない問題」になっており、導入の意図とずれている

- **箇所**: 352-451 行「Q3: setter! は表示される？」
- **指摘**: 3 問のクイズは「違和感を共有する」導入装置のはず。しかし Q3 の正解は `useDefineForClassFields` という固有名詞を知っていることが前提で、ジュニアどころか中堅でも初見で答えられる人は少ない。「クイズに答えて違和感を持つ」のではなく「答えがわからず置いてけぼり」になりかねない。
- **改善方針**: Q3 の選択肢を「A. 親 setter が呼ばれる / B. 親 setter は呼ばれない / C. 環境によって変わる」に簡略化する（現状の C と意味は同じだが、固有名詞を出さない）。「tsconfig の設定で変わる」という固有名詞の答え合わせは、解答スライドで初出にする。

### P3-2 🟡 ES4 挫折スライドが専門用語の洪水

- **箇所**: 531-549 行「ES4 の挫折 (2008)」
- **指摘**: 1 枚に「TC39」「Microsoft」「IE」「ブラウザ戦争」「Flash」「Silverlight」「JScript」「.NET ランタイム」「Harmony」「CoffeeScript」と固有名詞が 10 個。ジュニアにとっては「歴史の名前を覚える時間」になってしまう。中心メッセージ「ES4 が一度死んで、その間に class への渇望が溜まった」が固有名詞で埋もれる。
- **改善方針**: 固有名詞を 5 個以下に絞る。Silverlight / JScript / Macromedia 周りの説明は思い切って削り、「ES4 は野心的すぎて 2008 年に廃棄。その間に CoffeeScript などの altJS が class を提供して人気を集めた」程度に圧縮する。

### P3-3 🟡 `[[Define]]` vs `[[Set]]` の表記が初出のまま使われている

- **箇所**: 634-682 行「`[[Define]]` vs `[[Set]]` 論争」
- **指摘**: `[[Define]]` という二重角括弧表記は ECMAScript spec の内部メソッド記法だが、ジュニア聴衆にはまず通じない。スライドにも説明がない。コメントで「Object.defineProperty 相当」「`this.x = value` 相当」とは書いてあるが、本文の見出しは表記のまま。
- **改善方針**: 見出しを「`[[Define]]`（定義）vs `[[Set]]`（代入）— ES の内部メソッド表記」のように補助線を足す。あるいは「`Object.defineProperty` vs 代入」と平易な見出しに置き換え、`[[Define]]` 表記は本文に閉じ込める。

### P3-4 🟢 prototype ベースの説明とメソッド呼び出し 2 ステップ分解は良いが、その間にジャンプがある

- **箇所**: 898-921 行「なぜ this は動的に束縛されるのか」
- **指摘**: `obj.method(arg)` の 2 ステップ分解は素晴らしい。ただし「class はこの仕組みの上に載っている」直下で `Class.prototype` の話に飛ぶ。`Class.prototype` を見たことがない佐藤には「class なのに prototype？」というジャンプがある。
- **改善方針**: 「class 構文の裏側は prototype（前章 600 行の "コンパイル後 (ES5)" の図を再掲）」と図を再利用すれば、ジュニアもジャンプせずに済む。

### P3-5 🟢 Branded Type の `Symbol` 初出に説明がない

- **箇所**: 1077-1099 行「対策 1: Branded Types」
- **指摘**: `const UserIdSymbol = Symbol();` を `K & { [k in T]: true }` で使うパターンは初学者にとって型パズルの世界。「なぜ `Symbol` なのか」を一言ほしい。
- **改善方針**: 「`Symbol()` は呼ぶたびに必ず一意な値を返すので、衝突しない `ブランド` として使える」を一言足す。

## P4. 朝倉さん（Rust/Haskell, 仕様厳密性）視点

### P4-1 🔴 「2 つの根本原因」の不正確さ（P1-1 と重複するが角度が違う）

- **箇所**: 790-808 行、985-1011 行「落とし穴 4: instanceof と型消去」
- **指摘**: P1-1 の通り this は別原因。さらに instanceof の問題も厳密には「interface に runtime 実体がない」ことと「prototype チェーンの参照同一性が構造的部分型と噛み合わない」の 2 点で、「型消去」一語で片付けるのは粗い。仕様の議論として「型消去で全部説明できる」と言い切ると、Rust/Haskell からこの場に座っている朝倉には「説明が雑だ」と映る。
- **改善方針**: 「2 つ（または 3 つ）の根本原因」の章で、各落とし穴がどの原因にどの程度寄与するかを丁寧に切り分けるマトリクス図にする。

### P4-2 🔴 Branded Type の `Symbol()` 実装が `unique symbol` でないため、厳密には型レベルで分岐しない

- **箇所**: 1077-1091 行
- **指摘**: `const UserIdSymbol = Symbol();` の型は `symbol`（not `unique symbol`）。`Brand<K, T extends symbol>` の T に渡される `typeof UserIdSymbol` も `symbol`。これでは `UserIdSymbol` と `ProductIdSymbol` が型レベルで区別されない可能性がある。実際 TypeScript 5.x では `const x = Symbol()` の型は `symbol` で widening される（`declare const` でないと `unique symbol` にならない）。スライドのコード例は本当に動くか怪しい。
- **改善方針**: `declare const UserIdSymbol: unique symbol;` に修正するか、Zod の `.brand()` を使う方の例を主例に据える。または最近主流の `type UserId = string & { readonly __brand: "UserId" }` 方式に置き換える。**この修正はコード正確性に関わるため必須**。

### P4-3 🟡 ES4 挫折の歴史記述が一面的

- **箇所**: 531-549 行
- **指摘**: 「Microsoft が反対」だけでなく、Mozilla 側にも Brendan Eich vs Mark Miller の構造があり、Douglas Crockford も慎重派だった。さらに「JScript 言語が動作する .NET ランタイムとの相互運用性を重視し反対」という表現は JScript.NET のことを指していると思われるが、文として不明瞭。一次資料（ECMA 公開アーカイブの議論ノート）に当たる朝倉のような聴衆には荒く見える。
- **改善方針**: 文を「Microsoft と一部メンバーが互換性と複雑化を理由に反対」と単純化するか、複数勢力を中立的に併記する。少なくとも JScript と .NET の表現は精査する。

### P4-4 🟡 `useDefineForClassFields` の「ES2022 以上でデフォルト true」の条件記述が不正確

- **箇所**: 676 行
- **指摘**: 正確には「`target` が `ES2022` 以上、かつ `useDefineForClassFields` が未指定の場合にデフォルト `true`」。スライドの「target: ES2022 以上ではデフォルトで true」は伝わるが厳密ではない。
- **改善方針**: 「`target` が ES2022 以上のとき、未指定なら true」と一言補強する。

### P4-5 🟢 `maximally minimal classes` の主導者を Allen Wirfs-Brock 単独にしているのは強い

- **箇所**: 553-571 行
- **指摘**: TC39 公開資料を見ると Mark Miller, Erik Arvidsson, Allen Wirfs-Brock 等が議論を主導し、最終的に minimal クラス案がまとまった経緯がある。「Allen Wirfs-Brock が中心的に推進」は誤りではないが、強い言い方。
- **改善方針**: 「Allen Wirfs-Brock らが推進」と緩めるだけで朝倉は納得する。

## P5. 三浦さん（OSS, 型システム）視点

### P5-1 🔴 Branded Type 実装が 2026 年時点で古い（P4-2 と一部重複）

- **箇所**: 1077 行のコード例
- **指摘**: `type Brand<K, T extends symbol> = K & { [k in T]: true }` は 2018-2020 年頃の典型パターン。2026 年なら次のいずれかが妥当:
  - `type Brand<T, B extends string> = T & { readonly __brand: B }`（文字列ブランド、Effect-TS や io-ts でも採用）
  - `type Brand<T, B extends symbol> = T & { readonly [k in B]: true }` ただし B は `declare const X: unique symbol` で得る
  - Zod v4 の `z.string().brand<"UserId">()`
  ライブラリ作者の三浦が「これを自分の OSS に採用しようか」と検討する基準では、実装の安全性とエルゴノミクスの面で古い例は採用見送りになる。
- **改善方針**: P4-2 と同様の修正。最も無難なのは `T & { readonly __brand: "UserId" }` 方式で、`Symbol` を使う説明は補足扱いに格下げ。

### P5-2 🟡 `#private` の落とし穴を端折りすぎ

- **箇所**: 1173-1204 行「対策 3: private 修飾子 → ES `#private`」
- **指摘**: 「実行時にも強制される」「`any` キャストでも突破できない」のメリットは書かれているが、ライブラリ作者にとっての落とし穴（structuredClone でエラー、JSON.stringify で消失、Proxy/WeakMap で扱いづらい、継承時に親フィールドへ届かない）に触れていない。「公開ライブラリでは ES `#private` が安全」と推奨する以上、副作用も明示すべき。
- **改善方針**: 1196 行のテーブルに「Proxy / structuredClone / JSON.stringify との相性」の行を追加するか、ノートに 1 段落足す。

### P5-3 🟡 Stage 3 decorators の「採用判断」が示されていない

- **箇所**: 686-737 行
- **指摘**: 「TypeScript 5.0 で対応」「Legacy と互換性なし」までは書いてあるが、ライブラリ作者として「自分の OSS で Stage 3 を採用すべきか」の判断材料がない。Angular/NestJS は依然 legacy decorators 前提で、ts-loader/swc の対応状況や reflect-metadata との関係も決着していない。
- **改善方針**: 落とし穴/対策篇には組み込みづらいので、まとめ前か Appendix 的に「Stage 3 decorators を採用すべきタイミング」スライドを 1 枚足す。30 分の枠で厳しいなら、ノートに留めて口頭で触れる選択肢もある。

### P5-4 🟢 Zod の `.brand()` API がバージョン非明示

- **箇所**: 1107-1124 行
- **指摘**: Zod v3 と v4 で `.brand()` の API が違う（v4 では引数なしでも OK、ジェネリクスで指定する形が主流）。スライドのコードがどちらに準拠か明示されていないと、コピペ層が混乱する。
- **改善方針**: コード例の冒頭にバージョンコメント、または `import { z } from "zod"` の下に `// Zod v4` を付ける。
